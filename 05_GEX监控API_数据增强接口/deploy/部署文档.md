# gexmonitorapi 部署文档（AWS 轻量服务器 / Ubuntu）

> 目标读者：拿到本仓库、要把服务跑到 AWS Lightsail（Ubuntu）上的运维/开发者。
> 路线：**git clone + Python venv + systemd 常驻**，鉴权用单个 Bearer Token。
> 本文每一步都说明「做什么、为什么」，照着从上到下执行即可。

---

## 1. 服务概述与数据流

本服务把 GEX Monitor 公开分析页 BTC 的四个 tab 抓下来，结构化成一个 JSON 指标字典对外提供。

```
                       每 10 分钟后台刷新 / 手动 /v1/refresh
                                   │
   GEX Monitor 公开页  ──(无头浏览器渲染)──>  抓取层  ──(正则解析)──>  指标字典
   (gex/gamma/vol/flow)                    scraper.py        parsers.py
                                                │
                                          进程内缓存 + JSON 文件持久化 (cache.py)
                                                │
                              FastAPI + Bearer Token (app.py)  ──>  /v1/info
```

关键事实（决定了部署细节，务必理解）：

1. **页面是前端渲染的**：直接 GET 只能拿到 `加载中...` 骨架，指标值是浏览器里 JS 跑完才出现。
   所以服务器**必须能跑 headless Chromium**（这也是为什么要装浏览器和一堆系统库）。
2. **数据走 websocket 持续推送**：页面不会进入"网络空闲"状态，因此抓取**不等 network idle**，
   而是 DOM 加载后**固定等待约 12 秒**让面板渲染出来（由 `BROWSER_WAIT_MS` 控制）。
3. **抓取有三级兜底**：Scrapling 动态抓取（主）→ 原生 Playwright（兜底）→ 静态抓取（兜底）。
   只要服务器装好了 Chromium，主路径在 Linux 上通常直接成功。
4. **失败不清空旧数据**：某次抓取失败时保留上一次成功的缓存，并把 `stale` 标记为 `true`、
   在 `field_status` / `last_error` 里说明，绝不会用空值覆盖好数据。

## 2. 抓取的指标清单（共 19 个字段）

下表是服务对外交付的全部字段。括号内是页面上的真实标签来源。本地实测 **19/19 全部命中**

| | `dvol` | DVOL 波动率指数 | DVOL `42.9%` | `42.9` |
| | `market_state` | 市场 gamma 状态 | MM Short Gamma | `negative_gamma` |
| `gamma_exposure` | `n2` / `n1` | 下方关键负 gamma 位 | GAMMA COMPONENTS | `60000` / `65000` |
| | `flip_point` | Gamma 翻转点 | FLIP `67,371.951` | `67371.951` |
| | `volatility_trigger` | 波动触发位 | VOL TRIGGER | `65000` |
| | `spot_price` | 现货价 | SPOT PRICE | `66650.5` |
| | `magnet_price` | 磁吸位 | MAGNET | `70000` |
| | `p1` / `p2` | 上方关键正 gamma 位 | P1 / P2 | `80000` / `82000` |
| `volatility` | `iv_rv_ratio` | 隐含/已实现波动率比 | IV/RV RATIO `1.31x` | `1.31` |
| | `pcr` | 成交量 Put/Call 比 | PCR (VOLUME) | `0.96` |
| | `term_structure` | 各到期 ATM IV + 25Δ 偏斜 | Term Details 表 | 10 行数组 |

## 3. 接口契约

| 方法 | 路径 | 鉴权 | 说明 |
| --- | --- | --- | --- |
| `GET` | `/health` | 否 | 存活探针，返回 `{"status":"ok","version":"0.1.0"}` |
| `GET` | `/v1/info` | Bearer | 返回四组指标字典 + 抓取时间 + 缓存状态 + 缺失字段 |
| `POST` | `/v1/refresh?section=all` | Bearer | 立即重抓；`section` ∈ `gex_board\|gamma_exposure\|volatility\|flow\|all` |

鉴权：除 `/health` 外都要求请求头 `Authorization: Bearer <API_TOKEN>`，否则返回 `401`。

`/v1/info` 顶层元数据语义：
- `availability`：`ready`（全部命中）/ `partial`（部分缺失或有错误）/ `missing`（从未成功）。
- `stale`：上一次刷新是否发生过失败（失败会保留旧缓存）。
- `fetched_at`：最近一次抓取时间（UTC ISO8601）。
- `missing_fields` / `field_status`：逐字段说明哪些没拿到、原因是什么。
- `sections.<x>.last_error`：该分组最近一次抓取的错误信息（正常为 `null`）。

## 4. 前置条件

| 项目 | 要求 / 建议 | 说明 |
| --- | --- | --- |
| 实例规格 | **建议 2GB 内存（$12）**；最低 1GB 且必须加 swap | headless Chromium 比较吃内存，512MB 会 OOM |
| 系统镜像 | Ubuntu 22.04+/24.04 或 Debian 12（Bitnami 也可） | 本项目要求 **Python ≥ 3.11** |
| 出网 | 需能访问 `gexmonitor.com` 与下载浏览器内核 | 抓取与安装都要联网 |
| 端口 | 直连方式放行 TCP **8000**（建议限制来源 IP） | 也可走 Nginx 反代 80/443，见第 9 节 |
| 远端仓库 | 一个你能 push、服务器能 clone 的 Git 仓库 | 本文用 git clone 路线 |

> 系统自带 Python ≥ 3.11 即可（Ubuntu 22.04+/Debian 12/Bitnami 均满足）。更低版本（如 3.10）
> 需自行安装 3.11+，否则 `pip install -e .` 会因 `requires-python` 报错。

## 6. 第二步：在服务器上部署（SSH 登录后执行）

下面命令默认用 Lightsail 的默认用户 `ubuntu`，应用目录 `/home/ubuntu/gexmonitorapi`。
把 `<仓库URL>` 换成你的地址。

### 6.0 先确认环境
```bash
lsb_release -a        # 确认是 Ubuntu 24.04
python3 --version     # 确认是 3.12.x
```

### 6.1 系统依赖 + 克隆代码
```bash
sudo apt-get update
sudo apt-get install -y python3 python3-venv python3-pip git
git clone <仓库URL> /home/ubuntu/gexmonitorapi      # 默认分支 main
cd /home/ubuntu/gexmonitorapi
```
- `python3-venv`：建隔离虚拟环境用；`git`：拉代码用。

### 6.2 创建虚拟环境、装依赖、装浏览器
```bash
python3 -m venv .venv
.venv/bin/pip install --upgrade pip
.venv/bin/pip install -e .            # 安装服务本体及依赖（生产不需要 [dev]）

# headless Chromium 运行所需的系统库（用 root 跑，会按系统版本装对包）
sudo .venv/bin/playwright install-deps chromium

# 下载浏览器内核：Scrapling 主用 + 原生 Playwright 兜底，两者共用同一套 Chromium
.venv/bin/scrapling install
.venv/bin/playwright install chromium
```
**为什么三条浏览器命令都要**：`install-deps` 装的是操作系统层的 `.so` 依赖（字体、libnss 等）；
`scrapling install` / `playwright install chromium` 下载的是浏览器二进制本身。缺任意一个，浏览器都起不来。

### 6.3 （仅 1GB 实例）加 2GB swap
```bash
sudo fallocate -l 2G /swapfile && sudo chmod 600 /swapfile
sudo mkswap /swapfile && sudo swapon /swapfile
echo '/swapfile none swap sw 0 0' | sudo tee -a /etc/fstab
free -h
```
- 给 Chromium 留缓冲，避免抓取瞬间内存峰值把进程 OOM 掉。2GB 实例可跳过。

### 6.4 写配置文件
```bash
# 生成一个强 Token（复制输出备用）
python3 -c "import secrets; print(secrets.token_urlsafe(32))"

sudo cp deploy/gexmonitorapi.env.example /etc/gexmonitorapi.env
sudo nano /etc/gexmonitorapi.env        # 把 API_TOKEN 改成上面生成的值，保存
sudo chmod 600 /etc/gexmonitorapi.env   # 只让 root 可读，保护 Token
```
配置项含义见第 10 节。`/etc/gexmonitorapi.env` 用 600 权限是为了不让其他用户读到 Token。

### 6.5 安装并启动 systemd 服务
```bash
sudo cp deploy/gexmonitorapi.service /etc/systemd/system/
sudo systemctl daemon-reload
sudo systemctl enable --now gexmonitorapi      # 开机自启 + 立即启动
sleep 3 && sudo systemctl status gexmonitorapi --no-pager
```
- `enable --now`：既设为开机自启，又马上拉起来。
- 服务进程崩了会被 systemd 自动重启（`Restart=always`）。

### 6.6 放行端口（Lightsail 控制台）
实例页 → **Networking → IPv4 Firewall → Add rule** → 放行 **Custom TCP 8000**
（强烈建议把 Source 限制成你的固定出口 IP，减少暴露面）。

## 7. 验证与验收

首轮抓取在后台进行，四个 tab 各等约 12 秒渲染，**整体就绪需要 1～2 分钟**。期间 `/v1/info`
会返回 `availability: missing/partial`，属正常现象。

```bash
# --- 服务器本机 ---
curl -s http://127.0.0.1:8000/health; echo
curl -s -o /dev/null -w "no-token -> %{http_code}\n" http://127.0.0.1:8000/v1/info   # 期望 401

# 手动催一轮抓取（约 1 分钟返回），看是否拿到真实数值：
curl -s -X POST -H "Authorization: Bearer <TOKEN>" \
  "http://127.0.0.1:8000/v1/refresh?section=all" | head -c 600; echo

# 读取完整指标字典：
curl -s -H "Authorization: Bearer <TOKEN>" http://127.0.0.1:8000/v1/info | python3 -m json.tool | head -40
```

```bash
# --- 公网（从你自己的电脑）---
curl -s http://<实例IP>:8000/health
curl -s -H "Authorization: Bearer <TOKEN>" http://<实例IP>:8000/v1/info | head -c 600
```

**验收标准**：
- `/health` 返回 `status: ok`。
- 不带 Token 访问 `/v1/info` 返回 `401`。
- 带正确 Token 的 `/v1/info` 返回 `availability: ready`（或 `partial`），且 `gex_board`、
  `gamma_exposure`、`volatility`、`flow` 四组里有真实数值、`stale: false`。

## 9. （可选）Nginx 反代 + HTTPS

需要域名。先把 `deploy/gexmonitorapi.service` 里的 `--host 0.0.0.0` 改成 `--host 127.0.0.1`
（让 uvicorn 只听本机），`sudo systemctl daemon-reload && sudo systemctl restart gexmonitorapi`，再：

```bash
sudo apt-get install -y nginx
sudo cp deploy/nginx-gexmonitorapi.conf /etc/nginx/sites-available/gexmonitorapi
sudo nano /etc/nginx/sites-available/gexmonitorapi      # 改 server_name 为你的域名
sudo ln -s /etc/nginx/sites-available/gexmonitorapi /etc/nginx/sites-enabled/
sudo nginx -t && sudo systemctl reload nginx

# 域名解析到本机后，自动签发并配置 TLS：
sudo apt-get install -y certbot python3-certbot-nginx
sudo certbot --nginx -d 你的域名
```
启用 Nginx 后，把 Lightsail 防火墙的 8000 关掉，只留 80/443。

## 10. 配置项完整参考（`/etc/gexmonitorapi.env`）

| 变量 | 默认 | 说明 |
| --- | --- | --- |
| `API_TOKEN` | `dev-token-change-me` | **必须改**。所有 `/v1` 接口的 Bearer Token |
| `REFRESH_INTERVAL_SECONDS` | `600` | 后台自动刷新间隔（秒），默认 10 分钟 |
| `REQUEST_TIMEOUT_SECONDS` | `45` | 单次抓取超时（秒） |
| `CACHE_FILE` | `.cache/gexmonitorapi/cache.json` | 缓存持久化路径（确保运行用户可写） |
| `USER_AGENT` | `gexmonitorapi/0.1 ...` | 抓取使用的 UA |
| `ENABLE_BACKGROUND_REFRESH` | `true` | 是否开启后台周期刷新 |
| `REFRESH_ON_STARTUP` | `false` | 启动时是否阻塞式抓一轮（建议关，后台任务无论如何会先抓一次） |
| `BROWSER_WAIT_MS` | `12000` | DOM 加载后固定等待毫秒数，等面板渲染。**抓不到数据就调大** |
| `BROWSER_NO_SANDBOX` | `false` | 受限主机上 Chromium 起不来时设 `true`（加 `--no-sandbox`） |
| `BROWSER_NETWORK_IDLE` | `false` | 是否等网络空闲（本站长连 websocket，建议保持 `false`） |

> systemd 的 EnvironmentFile 格式：每行 `KEY=VALUE`，**不要加引号、不要写 export**。

## 11. 故障排查

| 现象 | 排查 / 解决 |
| --- | --- |
| `status` 报浏览器启动失败 / `Failed to move to new namespace` / Chromium 闪退 | Ubuntu 24.04 默认限制非 root 的 user namespace。**解法 A（推荐）**：`echo 'kernel.apparmor_restrict_unprivileged_userns=0' \| sudo tee /etc/sysctl.d/99-userns.conf && sudo sysctl --system && sudo systemctl restart gexmonitorapi`。**解法 B**：在 env 里加 `BROWSER_NO_SANDBOX=true` 后重启。 |
| `/v1/info` 全是 null | 后台首轮没跑完，等 1–2 分钟或手动 `POST /v1/refresh`；看 `sections.<x>.last_error`。 |
| `last_error` 含 `loading_placeholder` | 页面没渲染出来。调大 `BROWSER_WAIT_MS`（如 18000）后重启。 |
| `last_error` 含超时 / Timeout | 调大 `REQUEST_TIMEOUT_SECONDS`（如 60）后重启。 |
| 日志里有 `Killed` / 进程被 OOM | 内存不够。升到 2GB 实例，或确认第 6.3 步 swap 已生效（`free -h`）。 |
| `playwright install-deps` 不可用 | 等价命令 `sudo .venv/bin/patchright install-deps chromium`。 |
| 带正确 Token 仍 401 | 检查 env 里 `API_TOKEN` 是否有多余空格/换行；`curl` 里 Token 与 env 是否一致。 |
| 公网访问不通但本机通 | Lightsail 防火墙没放行 8000，或 systemd 里 `--host` 不是 `0.0.0.0`。 |

## 12. 安全与合规

- **鉴权边界就是 Bearer Token**：务必用强随机 Token，并把 `/etc/gexmonitorapi.env` 设为 600。
- **最小暴露**：直连方式下，Lightsail 防火墙的 8000 端口建议限制来源 IP；对外正式用建议走
  Nginx + HTTPS（第 9 节），避免 Token 明文走公网。
- **合规**：服务只抓取公开页面的可见内容，不抓登录/后台/非公开接口，也不二次包装官方
  `/api/` 数据接口；刷新间隔默认 10 分钟，避免高频请求。

## 13. 附录：目录与 systemd 单元

```
gexmonitorapi/
├─ src/gexmonitorapi/      # 服务代码（config/scraper/parsers/cache/app）
├─ tests/                  # 单元测试（pytest，13 项）
├─ deploy/
│  ├─ gexmonitorapi.service        # systemd 单元
│  ├─ gexmonitorapi.env.example    # 服务器环境变量模板
│  ├─ nginx-gexmonitorapi.conf     # 可选 Nginx 反代
│  ├─ README.md                    # 部署速查
│  └─ 部署文档.md                  # 本文档
├─ pyproject.toml
└─ README.md
```

systemd 单元关键字段（`deploy/gexmonitorapi.service`）：
- `User=ubuntu` / `WorkingDirectory=/home/ubuntu/gexmonitorapi`：以哪个用户、在哪个目录运行。
  浏览器内核装在该用户的 `~/.cache`，所以 `scrapling install` 必须用同一用户跑。
- `EnvironmentFile=/etc/gexmonitorapi.env`：从这里读取 `API_TOKEN` 等配置。
- `ExecStart=... uvicorn gexmonitorapi.app:app --host 0.0.0.0 --port 8000`：启动命令。
- `Restart=always` / `RestartSec=5`：崩溃自动重启。
- `TimeoutStartSec=90`：给首次拉起浏览器留足时间。

**卸载**：
```bash
sudo systemctl disable --now gexmonitorapi
sudo rm /etc/systemd/system/gexmonitorapi.service /etc/gexmonitorapi.env
sudo systemctl daemon-reload
rm -rf /home/ubuntu/gexmonitorapi
```
