推送审计流程优化方案

从 FMZ 长文本推送切换为“短摘要兜底 + 外部 Relay 完整审计”的双通道方案
日期
2026-06-17
适用范围
中性回路信号层 Signal Review Card 推送、邮件审计、网页留档
核心链路
信号生成 -> 写 signal_review.jsonl -> POST 完整 card 到外部 relay -> relay 发 HTML 邮件 / 保存网页 -> FMZ Log@ 发短摘要兜底
边界
不改方向判断、不改 EDB 权重、不改执行层下单逻辑；只优化审计投递与留档链路

1. 结论

当前实盘现象和本地 v1.2.1 注释共同说明：FMZ `Log @` 推送可作为即时提醒通道，但不适合承载完整信号审计正文。长文本会被截断，换行也可能被压平成单行，导致背景层、修正层、论证层、冲突层无法完整阅读。

因此，推荐将推送体系拆成两个通道：

  1. FMZ 短摘要兜底：继续使用 `Log(..., "@")` 或现有 `Log(text + " @")`，只发送 120-180 字的决策摘要和审计索引。
  2. 外部 Relay 完整审计：信号层生成完整 `Signal Review Card` 后，POST JSON 到自维护 relay。Relay 负责持久化、渲染 HTML 邮件、生成网页详情页，并返回 delivery_id / review_url。

这条方案把 FMZ 推送从“完整审计承载层”降级为“可靠触达提醒层”，让完整审计回到可控的外部投递系统中。

2. 依据与约束

2.1 FMZ 官方推送机制

  • FMZ `Log()` 输出内容以 `@` 结尾时，该日志进入消息推送队列，并按账户推送设置投递到邮箱、WebHook 等。
  • FMZ 实盘消息推送存在频率限制：同一实盘 20 秒周期内，只保留最后一条推送消息，其他推送会被过滤。
  • FMZ WebHook 推送采用平台推送设置中的 `{body}` 替换方式，官方示例为 GET URL，例如 `http://abc.com/push.php?data={body}`。
  • Python 策略里不使用 `HttpQuery()`；若要主动 POST 外部服务，应使用 Python 标准库 `urllib` 或同等可用 HTTP 客户端。

2.2 本地实盘证据

  • FMZ 邮件已成功到达，说明 `Log @` 触发链路可用。
  • 长正文在邮件中截断，且本地 `neutral_regulation_demo_fmz.py` v1.2.1 注释已记录：完整四层正文超过 FMZ 推送长度上限，观察到约 190 字处截断，换行被压平。
  • 当前代码已默认 `signal_review_push_compact=True`，说明短摘要是必要兜底，但它不能替代完整 Review。

3. 总体流程

1. 信号生成 NeutralRepair 进入确认态，EDB 生成结论、证据、冲突、阻断与因子截面。
2. 本地留档 先写 `signal_review.jsonl`，确保 relay 或邮件失败也不丢审计基准。
3. POST Relay 发送完整 card JSON，带 HMAC、timestamp、nonce、card_id 幂等键。
4. Relay 投递 持久化、渲染 HTML 邮件、生成网页详情页、记录 delivery 状态。
5. FMZ 兜底 发短摘要 `Log @`：结论、冲突、relay 状态、card_id / review_url。
信号层只负责生成与发送审计事件；完整阅读、邮件排版、网页留档全部交给 relay。

4. 组件职责

组件 职责 不负责
信号层 FMZ 策略 生成完整 `Signal Review Card`；先写 `signal_review.jsonl`；异步或短超时 POST relay；发送 FMZ compact 摘要。 不渲染复杂 HTML；不直接管理邮件账号；不阻塞主循环；不改变交易判断。
Relay API 接收完整 JSON；校验签名；幂等去重；持久化；触发邮件和网页渲染；返回投递状态。 不参与方向判断；不改写卡片事实；不持有交易所 API key。
审计存储 保存 raw JSON、HTML 快照、delivery log、错误状态；支持按 `card_id` 查询。 不作为交易状态机；不覆盖 FMZ 本地 JSONL。
邮件服务 发送 HTML 邮件，主题前置结论，正文按背景 / 修正 / 论证 / 冲突四层展示，可附 JSON。 不承载实时执行命令，不依赖邮箱回复触发交易。
FMZ Log@ 作为低延迟提醒和失败兜底，发送短摘要及 relay 状态。 不再承载完整审计正文。

5.1 Relay 返回

{
  "ok": true,
  "delivery_id": "mail-20260612-0001",
  "review_url": "https://audit.example.com/reviews/3e3b",
  "duplicate": false,
  "mail_status": "queued"
}

信号层不依赖 relay 返回来改变 EDB 结论。返回值只用于 FMZ compact 摘要和本地 `delivery_state` 记录。

6. 邮件与网页样式

6.1 邮件主题

[中性回路信号审计] BTC #3e3b WAIT_CONFIRMATION conf38 EDB+0.18

6.2 邮件正文结构

  1. 首屏摘要：结论、置信、是否可交易、最大同向证据、最大反向证据、解除条件。
  2. 背景层：NR 窗口、Anchor、TMV 24h / 48h、Macro 原始组件。
  3. 修正层：Gamma regime、net gamma、pin / flip、SRD、Funding。
  4. 论证层：EDB score、confidence decomposition、证据贡献账本。
  5. 冲突层：同向、反向、降权、阻断、解除条件。
  6. 附件 / 链接：raw JSON 附件或 review_url。

6.3 网页详情页

网页可比邮件更完整，建议支持：按卡号查询、按日期过滤、复制 JSON、导出 PDF、查看 delivery log、查看原始 `factor_cross_section`。

7. FMZ 短摘要格式

FMZ `Log @` 只保留最小可用信息，目标是在截断情况下仍能读到结论。

【信号审计】BTC #3e3b WAIT_CONFIRMATION 暂不交易 conf38 EDB+0.18 一致62% 覆盖83% 同向TMV/CVD 反向MACRO/SRD relay=OK mail=queued 详情 https://audit.example.com/reviews/3e3b

若 relay 失败，FMZ 摘要改为：

【信号审计】BTC #3e3b WAIT_CONFIRMATION conf38 relay=FAILED 本地已写signal_review.jsonl 等待重试

8. 故障降级策略

故障 系统行为 操作员看到什么
relay 超时 本地 JSONL 已写；事件进入 outbox；FMZ 摘要标 `relay=PENDING` 或 `FAILED`。 知道有信号，但完整邮件暂未送达，可查本地 JSONL。
relay 收到重复 card 按 `idempotency_key` 去重，返回 `duplicate=true`，不重复发邮件。 不收到重复邮件。
邮件服务失败 relay 保存 card 和网页；delivery log 标 `mail_failed`；可重试邮件。 FMZ 摘要可带 review_url；网页仍可查。
FMZ 推送截断 设计上只发送短摘要，截断也保留 card_id 和核心结论。 用 `card_id` 或链接进入完整页面。
网络全部不可用 本地 `signal_review.jsonl` 仍为最后审计基准；outbox 后续补发。 等待网络恢复后补发邮件，期间从 FMZ 日志 / 本地 JSONL 查。

9. 安全与合规边界

  • 最小密钥：relay token 只允许写入审计事件，不允许读历史、不允许触发交易。
  • HMAC 签名：签名内容包括 timestamp、nonce、card_id、body hash，防止伪造和重放。
  • TLS：relay 只开放 HTTPS，拒绝明文生产请求。
  • 幂等：`idempotency_key` 由 asset、card_id、confirmed_time 组成。
  • 敏感数据：不发送交易所 API key、账户余额、真实仓位明细；执行层信息只在未来明确授权后接入。
  • 审计可追溯：relay 保存 raw JSON、渲染 HTML、邮件状态、请求来源、签名校验结果。

10. 三种方案对比

方案 优点 问题 结论
继续 FMZ 长文本推送 改动最少，现有链路已通。 长正文截断、换行压平、20 秒推送覆盖、不可控长度上限。 不适合作为完整审计通道。
FMZ WebHook 直接承载全文 使用 FMZ 推送设置即可接入外部服务。 仍依赖 FMZ push body；官方示例为 GET `{body}` 替换，长中文 JSON 不稳。 只适合短触发，不适合完整 card。
策略直接 SMTP 发邮件 不需要中间服务。 凭据暴露、TLS/端口/超时麻烦、阻塞策略主循环、难以统一留档和重试。 不推荐。
POST 外部 relay，由 relay 发邮件 / 保存网页 完整 JSON 可控、邮件样式可控、重试/幂等/网页留档可控、FMZ 只做短摘要。 需要维护一个轻量服务。 推荐主方案。

11. 分阶段落地计划

P0 - Relay 最小闭环

  • 实现 `/v1/signal-review` POST 接口。
  • 校验 HMAC、timestamp、nonce。
  • 保存 raw JSON 到 SQLite / 文件 / 对象存储。
  • 返回 `delivery_id` 与 `review_url` 初始值。
  • FMZ compact 摘要显示 `relay=OK/FAILED`。

P1 - HTML 邮件

  • 接入 SES / Resend / SMTP 中任一邮件服务。
  • 渲染四层审计 HTML 邮件。
  • 主题前置 `asset / card_id / support_label / confidence / EDB`。
  • 支持失败重试与 delivery log。

P2 - 网页详情页

  • 为每个 card 生成只读详情页。
  • 支持按卡号查询和复制 JSON。
  • 邮件和 FMZ 摘要都带 `review_url`。

P3 - 补发与监控

  • 信号侧维护 outbox，网络失败后补发。
  • relay 增加投递状态仪表盘。
  • 监控：relay 延迟、邮件失败率、重复率、签名失败率。

12. 验收标准

  1. 实盘产生信号后，本地 `signal_review.jsonl` 必定先写入完整 card。
  2. relay 能收到完整 JSON，且字段包含 `conclusion / reasoning / conflict / blocking / factor_cross_section`。
  3. FMZ 收到短摘要，即使被截断也能看到 `asset / card_id / support_label / confidence`。
  4. HTML 邮件能完整展示背景层、修正层、论证层、冲突层。
  5. 重复发送同一 card 不产生重复邮件。
  6. relay 或邮件失败时，FMZ 摘要明确显示失败状态，本地 outbox 可重试。
  7. 邮件正文中的置信仍标注为“未校准证据质量分”，不误导为胜率。

13. 推荐最终形态

最终主链：信号层只做事实生成、结构化留档和短超时上报；relay 做完整审计投递、网页留档和重试；FMZ `Log @` 只保留短摘要兜底。这样既不破坏信号层只读边界，也不把完整审计绑死在 FMZ 的短推送机制上。
