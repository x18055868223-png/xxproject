# 信号审计推送格式重设计 v1.2

> 2026-06-22 r3.0 校准：早期样例里的“未校准”展示已退役。`confidence_calibration` 继续保留在 JSONL / 卡片原始字段中，但 FMZ 短推、邮件首屏和验收清单只说明“置信是内部证据质量分，不是胜率”，不再把未校准状态作为可见标签。

> 日期：2026-06-17
> 范围：信号层 `render_review_card_push(card)` 的邮件 / FMZ App 推送正文样式设计。
> 目标：不改变信号判断、不改 EDB 权重、不改执行层契约，只重设计审计推送文本，让操作员可以直接基于邮件完成一次有效 Review。

## 1. 背景与当前问题

2026-06-12 实盘推送已经证明 FMZ `Log(... + " @")` 推送链路可用，但当前正文样式仍不满足审计用途。实际收到的内容近似为：

```text
【中性回路·信号审计】2026-06-12 14:12:32
标的 BTC ｜ 现价 63,339.96 ｜ 卡号 #3e3b

◎ 结论
方向中性、暂不交易、等待确认，置信 38/100。

◎ 发生了什么
DIE+Anchor 时序窗口刚确认修复（位移向下、峰-0.92、合并4）。
方向层 5 项因子参与、整体看多（存在分歧）：量价主干多(+1.00)、期权偏斜空(-0.20)、宏观空...
```

### 1.1 主要弊端

1. **信息被截断后不可复盘**：邮件客户端横向展示时，正文变成一条长句，后半部分被截断；即使打开邮件，也缺少明确的数值表。
2. **重点不明**：标题和结论只说明“中性 / 暂不交易”，没有回答“为什么不是交易信号”“哪一个冲突最大”“下一次解除阻断看什么”。
3. **分层不清**：DIE+Anchor、TMV、Macro、GGR、SRD、Funding 被揉成一段叙事，背景层、修正层、论证层、冲突层没有边界。
4. **原始关键数值缺失**：net gamma、gamma regime、TMV 24h / 48h、macro_score、macro 组件、SRD vote、Funding、agreement、coverage、confidence decomposition 没有以可扫描方式展示。
5. **冲突过程不透明**：只说“存在分歧”，没有列出同向证据、反向证据、剔除证据、降权原因和冲突比例。
6. **审计索引弱**：只给 `signal_review.jsonl #card_id`，但没有提示哪些字段是邮件摘要、哪些字段应去 JSONL 深挖。

### 1.2 设计约束

- FMZ 官方 `Log()` 推送语义为：日志消息末尾带 `@` 会进入消息推送队列。官方示例可见 `Log("Hello FMZ Quant !@")` / `Log("Hello, #ff0000@")`。本项目继续使用 v1.1.0 的稳健写法：`Log(text + " @")`，即正文最后保留独立 `@` token。
- 同一信号事件只发送一条推送，避免 FMZ 实盘推送队列在短时间内只保留最后一条时覆盖前文。
- 邮件正文要优先适配 163 邮箱这类“长行预览截断”的客户端：每行短、信息块清楚、不要依赖表格渲染。
- 推送正文是 `signal_review.jsonl` 的操作员摘要，不替代 JSONL 全量结构化留档。
- 信号层保持只读观察：推送样式不得改变方向、置信、阻断、下游执行字段。

## 2. 新推送的信息架构

新推送采用“1 个头部 + 4 个审计层 + 1 个复盘索引”的固定结构。

```text
【中性回路·信号审计 v1.2】BTC #3e3b
时间 2026-06-12 14:12:32 UTC+8 ｜ 现价 63,339.96
结论 中性 / WAIT_CONFIRMATION / 暂不交易
置信 38/100 ｜ EDB +0.18 ｜ 一致 62% ｜ 覆盖 83%
一句话：窗口已开，但 TMV 与部分流量偏多，SRD / Macro 反向，Gamma 不增信；证据冲突未消解，等待确认。

【一、背景层：窗口与市场底色】
NR 窗口：CONFIRMED｜位移 向下｜峰值 -0.92｜合并 4｜Anchor score 72｜ND -0.31
TMV：blend +0.42 多｜24h +0.31｜48h +0.49｜窗口冲突 False
Macro：score +0.23 温和逆风｜confidence 1.00｜VOLQ +x bps / DXY +x bps / US10Y +x bps

【二、修正层：期权与安全门】
Gamma：regime POSITIVE_PIN｜net_gamma +12.4M｜pin 64,000｜flip 62,800｜ggr_mult 0.80｜veto None
SRD：vote -0.20 空｜RR25D -x｜skew_slope -x｜term_slope +x｜data OK
Funding：last +0.012%｜effect mild｜hard_warning False

【三、论证层：EDB 合成账本】
score_raw +0.24 → ggr_mult 0.80 → EDB +0.18
confidence = strength 0.44 × agreement 0.62 × coverage 0.83 × gamma 0.80 = 38
证据贡献：
  + TMV        vote +1.00｜w 0.34｜贡献 +56%｜量价主干偏多
  + CVD_4h     vote +0.35｜w 0.18｜贡献 +13%｜短流确认偏多
  - SRD        vote -0.20｜w 0.18｜贡献 -8%｜期权偏斜偏空
  - MACRO      vote -0.50｜w 0.16｜贡献 -17%｜宏观逆风
  0 GGR_SPATIAL vote +0.00｜w 0.00｜贡献 0%｜区制不给方向票

【四、冲突层：为什么不是交易信号】
冲突 MATERIAL｜ratio 38%
同向：TMV / CVD_4h
反向：SRD / MACRO
剔除/降权：GGR 只做置信门；Funding 未达拥挤阈；缺失项不重分配权重
阻断：SOFT_GATE WAIT_CONFIRMATION｜解除条件：反向证据回落或同向证据继续增强，置信 ≥50

【复盘索引】
JSONL signal_review #3e3b｜snapshot includes factor_cross_section / reasoning / conflict / blocking
说明：置信为内部证据质量分，不代表真实胜率或盈亏概率。
```

## 3. 四层定义

### 3.1 背景层

回答“当前发生在什么市场状态里”。该层不直接给交易结论，而是给所有后续论证的上下文。

必须展示：

- `window.nr_state` / `window.is_active`
- `window.episode_direction`
- `window.peak_m_die`
- `window.event_count_merged`
- `window.anchor_score`
- `window.anchor_nd`
- `factor_cross_section.tmvf.direction`
- `factor_cross_section.tmvf.tmv_blend`
- `factor_cross_section.tmvf.tmvf_24h.tmv_final`
- `factor_cross_section.tmvf.tmvf_48h.tmv_final`
- `factor_cross_section.tmvf.window_conflict`
- `factor_cross_section.macro_pressure.macro_score`
- `factor_cross_section.macro_pressure.macro_regime`
- `factor_cross_section.macro_pressure.macro_data_confidence`
- `factor_cross_section.macro_pressure.components[]` 的核心 raw bps / tier

### 3.2 修正层

回答“有哪些期权 / Gamma / Funding 因素在修正或限制方向”。该层负责把卖方策略最敏感的非现货要素展开。

必须展示：

- `factor_cross_section.gamma_regime.regime`
- `factor_cross_section.gamma_regime.regime_strength`
- `factor_cross_section.gamma_regime.net_gamma_notional`
- `factor_cross_section.gamma_regime.pin_strike`
- `factor_cross_section.gamma_regime.flip`
- `factor_cross_section.gamma_regime.confidence_multiplier` 或 EDB `ggr_mult`
- `factor_cross_section.gamma_regime.veto_reason`
- `factor_cross_section.skew.vote` 或 EDB evidence 中 `SRD.detail`
- `factor_cross_section.skew` 中 RR / slope / expiry / data state 可用字段
- `factor_cross_section.funding.last_funding_rate`
- `factor_cross_section.funding.tmvf_funding_effect`

若某字段缺失，不能静默省略，应以 `N/A` 展示，并在“剔除 / 降权”行说明原因。

### 3.3 论证层

回答“EDB 如何从证据到结论”。这一层是审计价值的核心。

必须展示：

- `reasoning.edb_score_raw`
- `reasoning.edb_score`
- `reasoning.confidence_decomposition.strength`
- `reasoning.confidence_decomposition.agr_factor`
- `reasoning.confidence_decomposition.cov_factor`
- `reasoning.confidence_decomposition.ggr_mult`
- `reasoning.confidence_decomposition.conf_pre_veto`
- `reasoning.confidence_decomposition.confidence_final`
- `reasoning.evidence[]` 中每个有效证据的：
  - `key`
  - `vote`
  - `weight`
  - `eff_weight`
  - `contribution_pct`
  - `aligned`
  - `gloss_cn`
  - 关键 detail

证据行固定使用短行，不使用长句：

```text
+ TMV        vote +1.00｜w 0.34｜贡献 +56%｜24h +0.31 / 48h +0.49 / blend +0.42
- SRD        vote -0.20｜w 0.18｜贡献 -8%｜RR -x / slope -x / expiry 48h
- MACRO      vote -0.50｜w 0.16｜贡献 -17%｜score +0.23 / Mild Headwind
```

### 3.4 冲突层

回答“为什么当前不能行动，或者为什么可以行动但风险在哪里”。冲突层必须把等待、阻断、反向证据讲清楚。

必须展示：

- `conflict.level`
- `conflict.ratio`
- `conflict.aligned_keys`
- `conflict.dissent[]`
- `conflict.explanation_cn`
- `blocking.block_kind`
- `blocking.hard_veto`
- `blocking.soft_gates[]`
- `blocking.unblock_hint_cn`

冲突层要避免“存在分歧”这种不可执行表述，改成：

```text
冲突 MATERIAL｜ratio 38%
同向：TMV / CVD_4h
反向：SRD(-0.20, w0.18) / MACRO(-0.50, w0.16)
等待原因：WAIT_CONFIRMATION，置信 38 < 50，且反向证据合计 -25%
解除条件：SRD 回中性或 Macro 逆风回落；或 TMV/CVD 继续增强使 confidence ≥50
```

## 4. 邮件阅读体验规范

1. **第一屏必须完整回答三件事**：结论、置信、为什么暂不交易 / 为什么可交易。
2. **单行长度控制在 80 个中文字符以内**，避免邮箱预览和手机端截断。
3. **每一层用独立标题**，标题用中文序号，避免 emoji 和过多符号。
4. **数值优先，叙事跟随**：先给 `score / vote / weight / contribution`，再给中文解释。
5. **重要反向证据不能被省略**：任何 `aligned=False` 且 `eff_weight>0` 的证据必须出现在冲突层。
6. **缺失项显式写 N/A**：不允许因为字段缺失让操作员误以为该因子无冲突。
7. **隐藏校准标签**：`calibration_state` 保留在 JSONL / 卡片原始字段中，推送正文不展示“未校准”；结尾只提示置信不是真实胜率。
8. **只推一条**：新格式仍由 `render_review_card_push(card)` 生成单个字符串，再由 `fmz_push()` 添加末尾 ` @`。

## 5. 字段映射建议

| 推送区块 | 字段来源 | 说明 |
|---|---|---|
| 头部 | `card_id`, `confirmed_time`, `price`, `conclusion` | 直接决定邮件预览摘要 |
| 背景层 / NR | `window` | 说明信号窗口是否真正打开 |
| 背景层 / TMV | `factor_cross_section.tmvf` | 展示 24h / 48h / blend / window_conflict |
| 背景层 / Macro | `factor_cross_section.macro_pressure` | 展示 macro_score、regime、confidence、组件 |
| 修正层 / Gamma | `factor_cross_section.gamma_regime`, `reasoning.confidence_decomposition.ggr_mult` | 区分方向票、置信门、硬否决 |
| 修正层 / SRD | `factor_cross_section.skew`, `reasoning.evidence[key=SRD]` | 展示期权需求面反向或确认 |
| 修正层 / Funding | `factor_cross_section.funding`, `reasoning.evidence[key=FUNDING]` | 展示拥挤 / 反身性 |
| 论证层 | `reasoning.evidence[]`, `confidence_decomposition` | EDB 合成账本 |
| 冲突层 | `conflict`, `blocking` | 操作员 Review 的行动依据 |
| 复盘索引 | `card_id`, schema note | 指向 JSONL 全量卡 |

## 8. 实现建议

仅修改推送渲染，不改卡片结构：

1. 保留 `build_signal_review_card()` 输出结构不变。
2. 替换 `render_review_card_push(card)` 的正文组装逻辑。
3. 新增小 formatter，建议命名：
   - `_operator_headline(card)`
   - `_background_layer_lines(card)`
   - `_correction_layer_lines(card)`
   - `_reasoning_layer_lines(card)`
   - `_conflict_layer_lines(card)`
   - `_evidence_row(evidence_item)`
   - `_compact_macro_components(macro_pressure)`
   - `_missing_or_excluded_notes(card)`
4. 对所有可选字段使用 `N/A`，不要抛异常吞掉推送；现有 fallback 仍保留。
5. `fmz_push()` 不改，继续由它追加末尾 ` @`。

## 9. 验收标准

### 9.1 静态验收

- 推送正文含固定标题：
  - `【一、背景层：窗口与市场底色】`
  - `【二、修正层：期权与安全门】`
  - `【三、论证层：EDB 合成账本】`
  - `【四、冲突层：为什么...】`
- 每条推送只调用一次 `fmz_push()`。
- 正文末尾仍由 `fmz_push()` 追加 ` @`。
- 邮件正文不使用 Markdown 表格，不使用 HTML，不依赖等宽字体才可读。
- 单行建议长度小于 80 个中文字符；证据行可略长，但必须一证据一行。

### 9.2 内容验收

构造至少三类卡片样例：

1. `WAIT_CONFIRMATION`：必须显示等待原因、反向证据和解除条件。
2. `TRADE_SUPPORT_WEAK/STRONG`：必须显示为何可交易、主要同向证据和剩余风险。
3. `HARD_VETO`：必须显示 veto 原因、触发数值、置信归零过程。

每类样例必须断言正文包含：

- `TMV`
- `Macro`
- `Gamma`
- `SRD`
- `confidence =`
- `冲突`
- `阻断`
- `JSONL signal_review`
- 置信语义说明（不展示校准状态标签）

### 9.3 操作员验收

操作员只看邮件正文，应能回答：

1. 这是不是可交易方向？
2. 当前最大同向证据是什么？
3. 当前最大反向证据是什么？
4. Gamma 是增信、降信还是 veto？
5. Macro 的原始压力来自哪几个组件？
6. 为什么等，而不是交易？
7. 下一次解除等待需要看哪些数值变化？
8. 去 JSONL 深挖时用哪个卡号？

如果任一问题不能从邮件正文直接回答，则推送格式仍不合格。

## 10. 不做事项

- 不在推送中塞执行层选腿、报价、下单建议。
- 不把 `SignalEvidencePackage` 扩成全量因子包；桥仍只传 digest。
- 不引入第二条推送。
- 不把置信写成胜率。
- 不在邮件里展示完整 JSON，完整结构继续留在 `signal_review.jsonl`。
