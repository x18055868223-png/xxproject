# 信号层倾向性论增层优化思路参考

日期：2026-06-05

定位：本文是基于当前工程资产，对信号层“到期前倾向性论证层”的优化设计参考。目标不是立刻替换当前 EDB，也不是把执行层改造成方向判断器，而是给出一个更接近“贝叶斯 / 信息论推断”的清晰构造：在 24-72h 到期窗口内汇聚趋势、期货、期权 gamma、偏斜/期限结构斜率、波动率、成交量、宏观和执行可行性相关证据，压噪声、防跳变、保留全链路溯源，并能被执行层安全消费。

---

## 0. 当前结论摘要

当前 EDB 已经具备“贝叶斯味”的后验合成雏形：用 TMV、CVD、MACRO、FUNDING、SRD、GGR_SPATIAL 六类证据形成 `edb_score / confidence / agreement / coverage`。但它本质仍是“加权方向票 + 一致度/覆盖度调制”，还不是严格的信息论推断层。

主要缺口在三点：

1. 证据独立性假设偏强。TMV、CVD、SRD、GGR、Macro 很多时候会被同一个价格冲击或风险偏好冲击共同驱动，简单相加会高估信息量。
2. 置信含义容易被误读。当前 `confidence` 更像证据质量分，不应在未校准前被解释为真实胜率或净利润概率。
3. 执行层消费方式仍偏线性。`SIGNAL_CONFIDENCE -> preferred_delta -> signal_fit` 会把未校准的信号置信直接转成更激进的短腿 delta 偏好，容易放大方向判断误差。

建议构造一个新的“BIRL”参考层：

```text
BIRL = Bayesian Information Reasoning Layer
中文名：贝叶斯信息论倾向性论证增强层
```

它不是黑箱模型，而是一个可审计、可追溯、可校准的证据后验层。它可以作为 EDB v2 的设计方向，也可以先作为 EDB 外围的论证增强层并行输出。

## 1. 设计边界

### 1.1 信号层只回答两个问题

信号层仍保持当前工程的第一性：

```text
1. 窗口何时开？
2. 24-72h 到期前更倾向哪一侧？
```

信号层不选腿、不报价、不算保证金、不下单、不读账本、不做持仓后管理。

### 1.2 执行层只消费结论，不穿透信号内部

执行层仍只读取 `SignalEvidencePackage`，不直接读取 TMV、SRD、GGR、Macro 等信号层内部因子。执行层可以消费：

```text
support_label
side_hint
confidence / evidence_quality
stability_state
reject_state
pre_trade_context
```

但执行层不应重新组合这些因子来重判方向。

### 1.3 双后验，而不是一个混合大模型

推荐拆成两个后验：

```text
BIRL-A：方向后验
  属于信号层。
  回答：24-72h 到期前价格更倾向上方、下方，还是无方向？

BIRL-B：可执行后验
  属于执行层 / 风险层。
  回答：这个方向在当前 IV、VRP、盘口、成本、保证金、组合风险、退出路径下是否值得进入 PLAN？
```

这样可以避免一个常见错误：把“方向可能对”与“卖方结构可赚钱”混成同一个分数。

## 2. 当前资产映射

### 2.1 信号层已有因子

| 证据域 | 当前因子 | BIRL-A 中的角色 | 是否直接给方向 |
|---|---|---|---|
| 数据可用性 | External Gate | 数据质量门 | 否 |
| 时序窗口 | M-DIE / NeutralRepair / Anchor | 窗口激活与 TTL | 否 |
| 价格背景 | TMV-F 24h/48h | 方向 prior / 背景态 | 是，主干 |
| 期货 / 成交量流 | CVD 4h/12h + funding | 订单流与拥挤度修正 | 是，但需降相关 |
| 宏观背景 | MPF | 风险 regime / cap / veto | 弱方向或阻断 |
| 期权需求 / 斜率 | SRD 25D RR、偏斜斜率、期限结构斜率 | 衍生品需求侧方向票与波动率结构修正 | 是，但需内生折扣 |
| Gamma 区制 | GGR | 安全门 / 置信调制 / 小空间票 | 主要否决，弱方向 |
| 审计事件 | signal_events / review card | 论证截面归档 | 否 |

### 2.2 执行层已有因子在推断体系中的位置

执行层已有 21 类因子不应进入 BIRL-A 重判方向，但应进入 BIRL-B 或校准闭环：

| 执行域 | 因子 | BIRL 中的定位 |
|---|---|---|
| pricing_gate | VRP 窗口门、候选门、forward_vol_hurdle、BS pricer | BIRL-B 可执行后验的硬门；不判方向 |
| plan | leg_selection、plans、spm_sim | 候选结构质量、保证金释放、成本与计划排序 |
| order | execution 纪律 | 执行可行性与滑点纪律，不进方向 |
| ledger | ledger + EntryRiskAnchor | 入场锚、账本血缘、事后归因基础 |
| portfolio_risk | 组合预算、区制 sizing、回撤熔断 | 可执行后验与 size 上限 |
| position_manage | 止盈、末日 Gamma、时间退出 | 持仓后管理与前向标签来源 |
| hedge | 五变量、persistence、状态机、触界概率 | 持仓后风险恶化判断；不做入场方向 |
| attribution | 最小归因 | 校准 BIRL-A/B 的真实贡献 |
| replay | 全链净 PnL 回放 | Gate B：验证方向正确是否转化为扣成本净利润 |

结论：方向后验与可执行后验必须分层。VRP 和执行成本可以否决交易，但不能把一个弱方向“抬成强方向”。

## 3. BIRL-A：方向后验设计

### 3.1 总体数据流

```text
External Gate / freshness
        |
        v
EvidenceUnit 标准化
        |
        v
证据分簇 cluster
        |
        v
簇内降噪 + 相关性惩罚
        |
        v
log-odds 后验更新
        |
        v
熵 / 覆盖 / 稳定性 / veto 调制
        |
        v
support_label + side_hint + trace
```

### 3.2 EvidenceUnit 标准结构

每条证据都必须转成统一结构：

```text
EvidenceUnit:
  key                 # TMV / CVD_4H / SRD / MACRO / GGR ...
  cluster             # price_regime / flow_reflexivity / option_demand ...
  raw_value           # 原始值
  normalized_value    # 归一化后的 [-1,1]
  vote                # 有符号方向票 [-1,1]
  reliability         # 数据可靠度 [0,1]
  data_quality        # 新鲜度 / 缺字段 / 暖机情况 [0,1]
  horizon_match       # 与 24-72h 目标窗口匹配度 [0,1]
  causal_discount     # 相关性/内生性折扣 [0,1]
  llr                 # 转成 log-likelihood ratio 后的贡献
  reason_cn           # 中文解释
  trace               # 原始字段来源
```

重点是 `causal_discount`。它承认一个事实：不是所有同向证据都是独立因果信息。

### 3.3 证据分簇

建议分为六簇：

```text
price_regime
  TMV 24h / TMV 48h

flow_reflexivity
  CVD flow-only 4h / 12h
  funding crowded / fuel

option_demand
  SRD rr_z
  SRD delta_rr
  skew slope / term-structure slope

gamma_safety
  GGR regime
  GGR confidence_multiplier
  GGR spatial_vote
  GGR veto

macro_regime
  VOLQ / DXY / US10Y pressure

timing_quality
  NeutralRepair
  Anchor
  data freshness
```

### 3.4 后验更新公式

不建议继续把所有证据直接线性加权成一个分数。建议使用近似 log-odds 结构：

```text
prior_log_odds = f(TMV_24h, TMV_48h)

evidence_llr_i =
  vote_i
  * reliability_i
  * data_quality_i
  * horizon_match_i
  * causal_discount_i
  * scale_i

cluster_llr_c =
  clamp(sum(evidence_llr_i in cluster c), -cluster_cap_c, cluster_cap_c)

posterior_log_odds =
  prior_log_odds
  + sum(cluster_llr_c)
  - correlation_penalty
  + safety_modifier
```

再把后验转成方向分数：

```text
posterior_score = tanh(posterior_log_odds / score_scale)
```

其中：

- `posterior_score > 0`：倾向上行，执行层对应 put credit spread。
- `posterior_score < 0`：倾向下行，执行层对应 call credit spread。
- `posterior_score` 近 0：无方向，等待或阻断。

### 3.5 为什么 TMV 应作为 prior

TMV 是价格趋势、动量、量能的背景态。对 24-72h 这种窗口，它比单个 CVD 或单个 skew 读数更稳定，因此应作为 prior，而不是普通证据票。

建议语义：

```text
TMV = 背景先验，不是最终结论。
其他证据 = 对背景先验的修正、确认、折扣或否决。
```

这样比“六票同桌投票”更符合实际市场结构。

### 3.6 CVD 的拆分

当前 CVD 证据里包含 price confirmation，这会和 TMV 的价格趋势重复。

建议拆成：

```text
CVD flow-only:
  看主动流是否真实同向。
  可作为方向 llr。

CVD price-confirm:
  只提高 reliability，不作为独立方向 llr。
```

这样可以保留量价共振信息，同时避免价格路径被计两次。

### 3.7 Macro 的角色

Macro 不适合在 24-72h 内强判 BTC 短方向。它更适合作为：

```text
1. regime prior 的小修正；
2. 极端逆风的 hard veto；
3. 置信上限 cap；
4. 事后分层校准维度。
```

因此 Macro 的 `causal_discount` 应低于 TMV / CVD flow-only / SRD，并且不应在普通环境中把方向强行翻面。

### 3.8 SRD 的角色

SRD 是非常重要的期权需求信息，但它是内生读数：偏斜可能是市场已经移动后的需求结果，也可能是保护需求提前定价。

建议语义：

```text
SRD = 期权需求侧边际信息。
它可以增强或削弱方向后验，但必须带内生性折扣。
```

SRD 的最大价值不是“raw RR 是正还是负”，而是：

```text
rr_z 相对基线偏离
delta_rr 偏斜动量
vote_confidence
到期匹配度
OI / liquidity 质量
```

### 3.9 GGR 的角色

GGR 不应被当成方向因子主力。它的顺序应是：

```text
1. hard veto：强负 gamma 放大，直接阻断单边卖权；
2. confidence multiplier：正 gamma 可小幅增信，负 gamma 砍置信；
3. spatial_vote：仅正 gamma 钉住区给小空间票；
4. hedge/position risk：持仓后作为风险恶化变量。
```

这符合当前 GGR 因子文档的第一性，也能防止“gamma 符号 = 方向确定性”的误用。

## 4. 置信与熵

### 4.1 不再把 confidence 叫真实胜率

在没有前向标签校准前，`confidence` 应解释为：

```text
证据后验质量 / 熵降低程度 / 可解释稳定度
```

不能解释为：

```text
真实上涨概率
真实胜率
卖方净利润概率
```

### 4.2 建议置信公式

```text
p_like = sigmoid(posterior_log_odds)

entropy = -p_like*log(p_like) - (1-p_like)*log(1-p_like)
entropy_norm = entropy / log(2)

confidence_raw =
  100
  * (1 - entropy_norm)
  * coverage_quality
  * data_quality
  * stability_quality
  * calibration_quality
  * ggr_safety_multiplier
```

解释：

- 后验越接近 0.5，熵越高，置信越低。
- 证据越完整，覆盖越高。
- 数据越新鲜，质量越高。
- 状态越稳定，置信越高。
- 未校准时 `calibration_quality` 应小于 1，防止过度自信。
- GGR 可以砍置信或 veto。

### 4.3 coverage 不只看 present

当前 EDB 已经加入覆盖度，但 BIRL 应进一步区分：

```text
coverage_expected:
  本轮理论上应该有的证据。

coverage_present:
  实际有数据的证据。

coverage_informative:
  有数据且通过信息阈值的证据。

coverage_independent:
  扣除同源相关后的有效证据覆盖。
```

最终置信应使用 `coverage_independent`，而不是只看 present。

## 5. 抗噪与防跳变机制

### 5.1 状态机

建议 BIRL-A 输出一个稳定性状态：

```text
OBSERVE
  有读数，但窗口未开或数据不足。

BUILDING
  方向开始形成，但持续性不足。

STABLE
  后验方向、覆盖度、数据质量、窗口状态均满足要求。

DECAYING
  原方向仍在，但证据衰减或冲突上升。

INVALIDATED
  hard veto、窗口失效、方向翻转或数据失真。
```

执行层只允许：

```text
stability_state == STABLE
AND support_label in {TRADE_SUPPORT_WEAK, TRADE_SUPPORT_STRONG}
```

进入计划层。

### 5.2 方向迟滞

方向不能 tick 级翻转。建议：

```text
多 -> 空：
  必须先进入 NEUTRAL / DECAYING，
  再连续满足空方向阈值 N 个 tick 或 T 秒，
  才能变成 BEARISH_STABLE。

空 -> 多：
  同理。
```

如果出现 GGR veto、Macro hard block、数据全坏，则可以直接 `INVALIDATED`，这是安全例外。

### 5.3 Deadband

所有证据都应有死区：

```text
abs(normalized_value) < informative_abs
  -> 只记录 trace，不进入 llr。
```

这样可以减少微弱噪声对后验的影响。

### 5.4 低通平滑

不建议对所有原始因子重度平滑，因为会牺牲反转速度。建议只平滑最终后验：

```text
posterior_log_odds_ema =
  alpha * posterior_log_odds_now
  + (1 - alpha) * posterior_log_odds_prev
```

同时保留 raw posterior 用于审计。

## 6. 可溯源论证结构

每次输出都应能解释为一段中文论证，而不是只有数字。

建议输出：

```text
trace_summary_cn:
  当前 24-72h 倾向为偏空/偏多/无方向。
  背景先验来自 TMV 24h/48h。
  期货流 CVD 是否确认。
  期权 skew 是否同向。
  Gamma 区制是否允许单边卖权。
  宏观是否构成逆风或阻断。
  当前最大冲突来自哪些证据。
  最终为什么支持/等待/阻断。
```

示例结构：

```text
结论：偏空但仅弱支持。
背景：TMV 24h/48h 同向偏空，形成下行先验。
确认：CVD flow-only 有中等同向流，但 price confirm 与 TMV 同源，未重复计入方向。
期权：SRD 相对偏斜同向，但因基线样本偏少，贡献降权。
安全：GGR 处于 transition，未否决但不增信。
宏观：轻度逆风，仅作为小修正。
冲突：慢窗 CVD 不充分，覆盖度中等。
动作：窗口开且稳定性达标时，可给 TRADE_SUPPORT_WEAK；否则 WAIT_CONFIRMATION。
```

## 8. BIRL-B：可执行后验

BIRL-B 不属于信号方向层，但执行层需要它，否则方向正确也可能因为 IV 不贵、成本太高、流动性差、退出困难而无交易价值。

建议执行层形成：

```text
ExecutableEdgePackage:
  signal_package_id
  direction_bias
  vrp_window_gate
  candidate_vrp_gate
  liquidity_quality
  spread_fee_friction
  spm_margin_relief
  portfolio_budget
  exit_path_quality
  plan_rank_hint
  executable_support
  reject_reason_codes
```

入场条件：

```text
BIRL-A support PASS
AND BIRL-A stability PASS
AND VRP window PASS
AND candidate VRP PASS
AND portfolio budget PASS
AND execution friction PASS
AND approval PASS
```

注意：BIRL-B 只能否决或缩小，不得把 BIRL-A 的弱方向抬成强方向。

## 9. 执行层消费方式修正

### 9.1 不建议继续线性 confidence -> delta

当前执行层将 `SIGNAL_CONFIDENCE` 线性映射到偏好短腿 delta。优化后建议改为分档：

```text
UNCALIBRATED / WEAK:
  只允许低 delta 区间，偏防守。

UNCALIBRATED / STRONG:
  可以略提高 delta，但仍受 VRP edge、流动性、预算限制。

CALIBRATED / STRONG:
  才允许更进取的 delta 偏好。
```

### 9.2 重命名启发式胜率

`plan_win_rate = 1 - |short_delta|` 应改名为：

```text
delta_otm_proxy
```

`plan_ev` 应改名为：

```text
delta_proxy_ev
```

原因：它们不是历史胜率和真实期望值，只是 delta 近似下的排序代理。

## 10. 校准与验证

### 10.1 必需样本

正式校准需要：

```text
snapshots.jsonl >= 2-4 周
覆盖：
  正 gamma / 负 gamma / transition
  宏观顺风 / 逆风
  funding 拥挤 / 中性
  CVD 同向 / 背离
  SRD 冷启动 / 正常
  趋势 / 震荡 / 急跌 / 急涨
```

当前本工程没有可直接用于验证 BIRL 的真实 `snapshots.jsonl`，因此本文所有权重、公式、阈值都只能作为设计参考，不是已证最优。

### 10.2 真值标签

方向后验标签：

```text
signal_confirmed_ts 后 24h / 48h / 72h 的收益符号
以及到期窗口内最大不利偏移 / 最大有利偏移
```

卖方可执行标签：

```text
扣除 bid/ask、fee、保护腿、滑点、对冲、退出成本后的组合净 PnL
```

### 10.3 消融验证

必须做条件消融：

```text
TMV only
TMV + CVD flow
TMV + CVD + SRD
TMV + CVD + SRD + Macro
TMV + CVD + SRD + GGR
Full BIRL
```

每个新增簇必须证明边际信息量：

```text
IC 提升
Brier / logloss 改善
冲突场景误放减少
强信号净 PnL 分桶改善
```

如果某个证据簇 IC 近 0，必须降权或只保留为 trace。

### 10.4 相关性矩阵

需要统计：

```text
corr(TMV, CVD_price_confirm)
corr(TMV, SRD)
corr(TMV, Macro)
corr(SRD, GGR)
corr(GGR regime, realized_vol)
```

高度相关的簇不得线性叠加，必须触发 `correlation_penalty` 或 `cluster_cap`。

## 11. 推荐落地阶段

### Phase 0：文档与契约冻结

```text
确认字段 schema；
确认执行层只消费包，不穿透因子。
```

### Phase 1：并行影子输出

在信号层输出中新增 `bayesian_reasoning`，但不改变当前 EDB 交易支持结果。

```text

积累两者差异；
记录 trace。
```

### Phase 2：离线回放与校准

用真实 `snapshots.jsonl` 和前向标签校准：

```text
cluster weights
causal_discount

stability thresholds
```

### Phase 3：替代 EDB 分类

只有当 BIRL 在方向标签与扣成本净 PnL 两侧都优于 EDB，才允许：

```text
BIRL support_label 替代 EDB support_label
```

### Phase 4：执行层分档消费

执行层改为：

```text
confidence_quality + stability_state + calibration_state
```

分档影响计划偏好，而不是线性推高 delta。

## 15. 当前版本 Bug：信号推送未正确触发

### 15.1 现象

当前 1.0.0 信号层已有 Signal Review Card，并且 `_emit_signal_review_card()` 在 `signal_review_push_enabled=True` 时调用：

```text
fmz_push(render_review_card_push(card))
```

但用户实测“出信号后未能正确推送”。该问题应作为当前版本 bug 写入后续修复清单。

### 15.2 可能原因

最新版 `fmz_push(text)` 当前实现语义是：

```text
Log(str(text) + "@")
```

但 FMZ 文档推荐使用：

```python
Log("\n".join(lines) + " @")   # 末尾 @ 触发推送
```

差异点在于：

```text
当前：正文直接拼接 "@"
建议：正文末尾拼接 " @"
```

如果 FMZ 的推送解析对末尾空格或 token 边界敏感，`text + "@"` 可能只进入普通日志，不能稳定进入推送队列。另一个现实限制是 FMZ 20 秒内通常只推最后一条，因此同一信号事件内不能分多次推送。

### 15.3 推荐修复

不建议分开推送 EDB 与 BIRL。应当合并成一次推送：

```python
lines = []
lines.append("【中性回路·信号触发】")
lines.append("")
lines.append("【一、EDB 到期方向合成层】")
lines.extend(edb_lines)
lines.append("")
lines.append("【二、BIRL 并行论证增强层】")
lines.extend(birl_lines)
lines.append("")
lines.append("【三、数据质量与阻断项】")
lines.extend(quality_lines)
Log("\n".join(lines) + " @")
```

关键要求：

1. 只推一次，避免 FMZ 20 秒推送队列覆盖。
2. 末尾使用 `" @"`，与 FMZ 文档形式一致。
3. EDB 与 BIRL 用不同子标题区分。
4. BIRL shadow 阶段必须标注“并行参考，不改变 EDB 交易支持”。
5. 若 BIRL 不可用，则仍推 EDB 审计卡；若 EDB 阻断，也要明确阻断原因。

### 15.4 推荐推送结构

```text
【中性回路·信号触发】

【一、EDB 到期方向合成层】
结论：...
置信：...
证据：TMV / CVD / Macro / SRD / GGR ...
阻断/等待：...

【二、BIRL 并行论证增强层】
状态：shadow / uncalibrated
后验：...
熵：...
主要簇贡献：price_regime / option_demand / gamma_safety ...
相关性惩罚：...
解释：该层仅作并行论证，不改变当前 EDB 放行。

【三、GEX Monitor 与数据质量】
Gamma 体制：...
关键位：...
IV/RV / PCR / term slope：...
新鲜度：...
```

### 15.5 是否必须引入 BIRL 后再修

不必须。推送 bug 是独立 bug，应该优先修：

```text
先修推送格式：Log("\n".join(lines) + " @")
再考虑 BIRL 并行输出。
```

如果后续引入 BIRL，则在同一条 Signal Review 推送中并入 BIRL 子标题；不要新增第二条推送。

## 16. 与当前项目最匹配的推荐

推荐路线：

```text
先做 BIRL shadow，不立刻替换 EDB。
先增强论证与 trace，不立刻改变交易行为。
先校准方向后验，再校准执行后验。
先证明净 PnL 改善，再让执行层消费更进取档。
```

原因：

1. 当前 EDB v0.5.4 已经修复了很多低级问题，直接替换风险较高。
2. 项目当前最缺的不是更多分数，而是真实前向样本与扣成本回放。
3. BIRL 的价值在于把“相关性不等于因果”的问题显式编码进 `causal_discount / cluster_cap / correlation_penalty`。
4. 执行层需要的是稳定、可解释、可否决的前置信号，而不是更复杂但不可追溯的黑箱。

最终语义：

```text
EDB v0.5.4 = 当前权威方向层。
BIRL v0.1 = 倾向性论证增强层 / EDB v2 候选设计。
未完成真实校准前，BIRL 不应直接授权实盘。
```
