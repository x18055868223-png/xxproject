## 二、核心语义修正

### 2.1 止盈资格与止盈成交必须拆开

原设计把“可执行净利润捕获率”同时作为止盈触发和实际成交判断。对于已经深度虚值、盘口稀疏的卖方价差，这会产生一个实际问题：

- 组合已经实现了绝大部分经济利润；
- 卖方短腿与保护腿都可能缺少连续报价；
- 如果必须先得到一组当前可执行组合报价，系统可能永远无法进入止盈流程。

因此本版本改为：

```text
止盈资格触发
  = 参考价值表明利润捕获已达到阈值

实际止盈成交
  = 在利润保护支出预算内，持续进行低成本退出尝试
```

止盈资格只负责回答“现在是否值得开始退出”；实际成交预算负责回答“最多愿意花多少钱退出”。两者不再混为一个指标。

### 2.2 80% 阈值的新定义

入场完成后，根据真实成交冻结：

```text
entry_profit_ceiling_net
  = actual_short_credit
  - actual_long_debit
  - actual_entry_fees
```

默认止盈目标：

```text
take_profit_target_ratio = 0.80

target_profit_amount
  = entry_profit_ceiling_net * take_profit_target_ratio

max_total_exit_spend
  = entry_profit_ceiling_net - target_profit_amount
```

`max_total_exit_spend` 是本轮止盈活动允许消耗的最高总预算，至少包括：

- 买回卖方短腿的成交支出；
- 卖方短腿退出手续费；
- 撤单、重挂和预留滑点；
- 保护腿处置手续费预留。

为避免保护腿流动性差导致止盈无法启动，**保护腿回收价值在止盈预算中默认按 0 计算**。卖出保护腿所得属于额外回收，不作为保证 80% 利润捕获的前提。

### 2.3 止盈资格参考值

止盈资格可以使用不要求立即成交的保守参考值：

```text
reference_exit_spend
  = conservative_short_buyback_reference
  + estimated_short_exit_fee
  + exit_reserve

reference_profit_capture_ratio
  = (entry_profit_ceiling_net - reference_exit_spend)
    / entry_profit_ceiling_net
```

`conservative_short_buyback_reference` 的数据优先级：

1. 有效且新鲜的卖方短腿 mark；
2. 有效的独立定价参考；
3. 最近有效成交与 Greeks/IV 约束下的保守估值；
4. 数据不可用时不触发自动止盈资格，保持监控并标记数据缺口。

保护腿价值不进入触发分母，也不用于美化止盈比例。

### 2.4 为什么这种改法更适合当前结构

只要短腿可以在 `max_total_exit_spend` 内买回，即使保护腿最终无法卖出，80% 目标仍然不会因保护腿缺少 bid 而被破坏。

因此：

- 流动性差会影响退出完成时间；
- 流动性差不会阻止系统开始退出；
- 实际订单仍受硬支出预算约束；
- 不会为了追求成交而无限加价。

## 三、唯一结构：垂直信用价差

### 3.1 当前版本允许的结构

只允许：

```text
SHORT_CALL_VERTICAL
SHORT_PUT_VERTICAL
```

结构约束：

- 两条期权腿到期时间相同；
- 保护腿比卖方短腿更价外；
- 入场时先买保护腿，再卖短腿；
- 卖方短腿数量不得大于已成交保护腿数量；
- 退出时先买回卖方短腿，再处理保护腿；
- 不允许保护腿先退出后留下裸卖方腿。

### 3.2 必须从运行版本删除的日历路径

从 `demo/execution_build/realsrc` 的当前运行路径删除：

- `MODE_CALENDAR`
- `ENABLE_CALENDAR`
- 日历保护腿到期枚举
- 日历价差方案生成与推荐
- 日历保护腿复用与周期分摊
- 日历残值恢复相关排名语义
- 日历价差展示标签
- 同时验证垂直与日历的运行测试
- `CALENDAR_PROTECTED_SHORT_OPTION` 结构类型

保留历史资产不等于保留运行能力。当前版本的源码、测试、生成单文件和最新交付物中，不得再产生或推荐日历价差。

### 3.3 垂直价差推荐库

删除日历价差后，策略推荐库仍然必须维护。推荐库只包含通过全部门控的垂直价差方案：

```text
VerticalRecommendationLibrary
  session_id
  signal_package_id
  generated_ts
  refresh_seq
  recommendations[]
```

每个推荐项至少包含：

```text
strategy_code
quality_code
plan_hash
rank
recommendation_state
side
expiry
short_instrument / short_strike
long_instrument / long_strike
amount
entry_net_credit_after_costs
max_loss
spm_result
vrp_result
projected_budget
created_ts / expires_ts
```

推荐库持续刷新、持续排序、持续展示，但不会自动进入执行层。

## 四、目标主链与双交互模型

### 4.1 主状态机

```text
IDLE
  -> SIGNAL_OBSERVED
  -> PRICE_GATE
  -> RECOMMENDATION_LIBRARY_READY
  -> PLAN_HARD_APPROVAL_WAIT
  -> PLAN_LOCKED
  -> PRECOMMIT_RECHECK
  -> ORDER_COMMITTING
  -> OPENING_LONG_PROTECTION
  -> OPENING_SHORT
  -> POSITION_OPEN
  -> POSITION_MANAGE
  -> SHORT_EXIT_CAMPAIGN
  -> LONG_RECOVERY_CAMPAIGN
  -> CLOSED
```

并行子域：

```text
SIGNAL_REFRESH
MARKET_REFRESH
LEDGER_RECONCILE
RISK_EVALUATION
HEDGE_WATCH
EXIT_AUTHORIZATION
ORDER_RECOVERY
```

### 4.2 两种交互的本质差异

| 交互 | 性质 | 未获得交互时 | 获得交互后 |
|---|---|---|---|
| 计划进入执行 | 硬交互 | 绝对禁止进入执行层 | 仅允许指定方案进入预提交复核 |
| 持仓止盈/退出 | 软授权 | 不下退出单，但主循环全部正常运行 | 允许当前持仓在条件满足时启动自动退出活动 |

硬交互是状态机门；软授权是一个与主状态机并行的授权标志。软授权不能把程序阻塞在“等待用户”状态。

## 五、硬交互：计划层进入执行层

### 5.1 硬交互要求

计划层生成推荐库后，在收到与当前推荐项完全匹配的交互指令前：

- 不得锁定方案；
- 不得进入预提交；
- 不得创建开仓订单；
- 不得默认选择排名第一的方案；
- 不得通过旧 `SELECTED_PLAN` 或重启式 `ROUND_MODE` 绕过。

计划层可以继续：

- 刷新信号；
- 刷新行情；
- 更新推荐库；
- 更新方案排名；
- 展示拒绝理由；
- 维护历史推荐审计。

### 5.2 策略编码与质量编码

每个推荐项使用两个面向交互的不可混淆编码：

```text
strategy_code
  = 对结构身份的稳定编码

quality_code
  = 对当前质量快照的短编码
```

建议结构：

```text
strategy_code
  = VCS|BTC|CALL/PUT|EXPIRY|SHORT_STRIKE|LONG_STRIKE

quality_code
  = 当前 signal / VRP / SPM / quote / budget 快照哈希前缀
```

`strategy_code` 确认用户选择的是哪一个垂直价差；`quality_code` 确认用户批准的是哪一次质量快照。推荐项刷新后质量发生变化，`quality_code` 必须变化，旧交互自动失效。

### 5.3 FMZ 命令

硬交互命令：

```text
PLAN_EXECUTE:<session_id>|<strategy_code>|<quality_code>|<plan_hash8>
PLAN_REJECT:<session_id>
```

只有以下条件全部满足，`PLAN_EXECUTE` 才能创建锁定方案：

- `session_id` 与当前会话完全一致；
- `strategy_code` 存在于当前推荐库；
- `quality_code` 与当前推荐项一致；
- `plan_hash8` 与完整 `plan_hash` 一致；
- 推荐项未过期；
- 推荐项仍为 `QUALIFIED`；
- 当前结构为垂直价差。

任一不匹配时：

```text
保持 PLAN_HARD_APPROVAL_WAIT
记录拒绝原因
继续维护推荐库
不进入执行层
```

### 5.4 锁定后与推荐库的关系

正确命令到达后：

1. 从推荐库复制一份不可变的 `LockedVerticalPlan`。
2. 锁定计划不再随推荐库刷新而变化。
3. 推荐库继续刷新，用于展示、审计和下一轮计划。
4. 锁定计划进入预提交复核。
5. 预提交复核失败则撤销本次锁定，返回推荐库等待状态；不得自动替换为其他推荐项。

## 六、软交互：持仓止盈与退出授权

### 6.1 软授权可以做到，而且应当这样设计

软授权不能实现成主状态机中的阻塞等待节点。它应当是当前持仓上的独立授权合同：

```text
PositionExitAuthorization
  position_id
  authorization_state
  policy_code
  authorized_ts
  revoked_ts
  authorization_hash
  operator_note
```

状态：

```text
UNAUTHORIZED
AUTHORIZED
REVOKED
CONSUMED
```

授权与当前 `position_id` 绑定。默认持续到：

- 用户撤销；
- 当前持仓完全结束；
- 账本或持仓身份发生无法解释的变化。

它不使用像计划授权那样的短 TTL，因为它授权的是“本轮持仓可以被自动退出”，不是授权某一瞬间的报价。

### 6.2 FMZ 命令

```text
EXIT_AUTHORIZE:<position_id>|BOUNDED_EXIT_V1|<authorization_hash8>
EXIT_REVOKE:<position_id>
```

收到 `EXIT_AUTHORIZE` 后：

- 不立即下单；
- 不停止风险评估；
- 不停止信号、行情和数据刷新；
- 不停止记账与对账；
- 不停止对冲监控；
- 只允许退出执行器在满足规则时工作。

未授权时：

- 继续计算止盈资格；
- 继续输出 `TAKE_PROFIT_READY` / `EXIT_PREFERRED`；
- 继续记录理论退出预算和风险；
- 不创建真实期权退出订单；
- 其他全部流程正常运行。

### 6.3 授权后的非阻塞退出子流程

软授权与退出活动分别记录：

```text
authorization_state = AUTHORIZED

exit_campaign_state
  = IDLE
  | WAIT_TRIGGER
  | WORKING_SHORT
  | WORKING_LONG
  | PAUSED_BY_DATA
  | PAUSED_BY_BUDGET
  | COMPLETE
```

主循环每轮只给退出活动有限处理时间，之后继续运行其他域。不得使用一个长循环阻塞行情刷新、风险评估或账本维护。

## 七、低成本止盈退出活动

### 7.1 启动条件

`SHORT_EXIT_CAMPAIGN` 仅在以下条件同时成立时启动：

- 当前持仓已有有效 `PositionExitAuthorization`；
- `reference_profit_capture_ratio >= 0.80`，或风险裁决为 `EXIT_PREFERRED`；
- 账本与交易所事实一致；
- 当前没有身份不明订单；
- 当前卖方短腿数量大于 0；
- 可计算本轮退出支出预算；
- 退出不会违反数量和结构不变量。

### 7.2 止盈退出的硬预算

对普通止盈：

```text
short_buyback_budget
  = max_total_exit_spend
  - realized_exit_spend
  - remaining_fee_reserve
```

卖方短腿任何新订单必须满足：

```text
order_price * order_amount + estimated_fee
  <= remaining short_buyback_budget
```

如果无法在预算内成交：

- 不突破预算追价；
- 撤销未成交订单；
- 保持授权；
- 继续监控并在后续轮次重新尝试；
- 状态记为 `PAUSED_BY_BUDGET`，而不是错误地宣称退出完成。

### 7.3 反复步长询价与挂单

现有建仓函数只允许有限的一步追价，不适合处理虚值期权的低成本退出。本版本新增专用的 `ExitCampaignExecutor`，不能直接复用建仓追价参数。

每个卖方短腿退出尝试：

1. 刷新 ticker、order book、mark、最后成交和 tick size。
2. 计算当前被动买入价格。
3. 价格不得超过本轮剩余支出预算对应的硬上限。
4. 下达带唯一 label 的 post-only 买单。
5. 等待一个短询价周期。
6. 查询订单最终状态。
7. 对未成交部分撤单。
8. 撤单后再次查询最终成交，处理取消竞态。
9. 更新真实成交、费用、剩余数量和剩余预算。
10. 下个主循环再决定是否提高一个 tick 或继续等待。

价格阶梯：

```text
passive start
  -> 每轮最多改善一个 tick
  -> 始终低于或等于 short_buyback_price_cap
  -> 始终服从 post-only
  -> 不因退出活动持续时间长而突破预算
```

每轮的价格上限应直接由剩余预算反推：

```text
short_buyback_price_cap
  = floor_to_tick(
      (remaining_short_buyback_budget - fee_reserve)
      / remaining_short_qty
    )

next_passive_price
  = min(
      maker_safe_price_from_current_book,
      short_buyback_price_cap
    )
```

如果缺少可靠盘口，系统可以继续刷新和记录资格，但不得用失真的 mark 直接创建订单。存在有效 best bid/ask 时，订单必须保持 post-only；只有独立风险退出规则才可以授权更激进的成交方式。

必要配置：

```text
EXIT_QUOTE_REFRESH_MS
EXIT_ORDER_REST_MS
EXIT_REPRICE_COOLDOWN_MS
EXIT_MAX_ACTIVE_ORDERS = 1
EXIT_MAX_PRICE_STEPS_PER_LOOP = 1
```

退出活动可以跨多个主循环和较长时间持续，但每轮只做一次有限动作。

### 7.4 先平卖方腿

退出顺序是硬不变量：

```text
卖方短腿未完全归零
  -> 禁止卖出保护腿

卖方短腿完全归零
  -> 才允许进入 LONG_RECOVERY_CAMPAIGN
```

卖方短腿部分成交时：

- 账本更新剩余短腿数量；
- 保护腿全部保留；
- 对冲目标按剩余短腿风险重新计算；
- 不提前卖出“多余”的保护腿。

此规则牺牲少量资金效率，换取更清晰的退出安全语义。

### 7.5 再平买方保护腿

卖方短腿归零后，启动保护腿回收活动：

```text
LONG_RECOVERY_CAMPAIGN
```

处理顺序：

1. 刷新保护腿报价。
2. 以被动 maker 卖单开始。
3. 按配置逐 tick 下调，但不得接受负净回收。
4. 每轮撤单、终态复核、记账。
5. 无 bid 或净回收过低时，保持 `LONG_RESIDUAL_ONLY` 并继续后续尝试或等待结算。

保护腿无法成交不影响卖方风险已经解除的事实，但：

- 持仓历史必须记录其仍存在；
- 会话不能伪装成两腿均已成交关闭；
- 保护腿出售或结算后才能完成最终账本归档。

### 7.6 风险退出与止盈退出的预算边界

软授权允许退出执行器工作，但不能把普通止盈预算悄悄升级为无限风险退出预算。

区分：

```text
TAKE_PROFIT_EXIT
  -> 严格服从 80% 利润保护预算

RISK_EXIT
  -> 使用独立配置的风险退出最大支出
```

当风险恶化且止盈预算无法成交时：

- 系统继续输出 `EXIT_PREFERRED`；
- 如果 `ALLOW_HEDGE_TRADING=True`，可按对冲规则压缩风险；
- 是否允许更高成本退出由独立风险退出预算决定；
- 不得使用普通止盈授权作为无限追价许可。

## 八、正确开仓与预提交

### 8.1 硬交互不是下单命令

正确的 `PLAN_EXECUTE` 只允许指定计划进入预提交复核，不直接调用订单 API。

预提交必须重新检查：

```text
signal_fresh
same_signal_package
locked_plan_hash_match
locked_quality_code_match
vertical_only
vrp_rechecked
spm_rechecked
quotes_rechecked
entry_net_credit_after_costs_positive
projected_budget_passed
ledger_reconciled
no_unknown_orders
spread_ok
```

任何失败都返回计划层，不自动改选推荐库中的其他方案。

### 8.2 开仓顺序

```text
先买保护腿
  -> 按真实保护腿成交数量限制卖方短腿数量
  -> 再卖卖方短腿
```

开仓订单继续采用：

- post-only；
- 唯一 order label；
- pending intent 先落盘；
- 有限步长；
- 撤单后最终状态复核；
- 部分成交记账；
- 保护腿成交、短腿无法成交时的受控回退。

### 8.3 成交后冻结的入场快照

```text
VerticalEntrySnapshot
  position_id
  session_id
  signal_package_id
  strategy_code
  quality_code
  plan_hash
  short_fill
  long_fill
  entry_fees
  entry_profit_ceiling_net
  take_profit_target_ratio
  max_total_exit_spend
  entry_risk_anchor
  vrp_lineage
  projected_budget_snapshot
```

入场后禁止重新计算或覆盖 `entry_profit_ceiling_net`，否则 80% 阈值会失去审计基准。

## 九、持仓管理主循环

### 9.1 每轮必须运行的域

无论是否获得软退出授权，每轮必须继续：

1. 刷新期权与 `BTC-PERPETUAL` 行情。
2. 读取并核对交易所真实持仓。
3. 维护订单和成交状态。
4. 更新期权账本。
5. 更新对冲账本。
6. 计算止盈资格。
7. 计算触界概率、漂移、斜率和尾部加速。
8. 计算退出摩擦与对冲摩擦。
9. 更新风险裁决。
10. 输出状态、原因与审计记录。

软授权只影响“退出执行器是否可以创建订单”，不影响以上任何工作。

### 9.2 统一动作优先级

每轮只允许一个最高优先级风险动作：

```text
1. 订单或账本身份不明
   -> RECOVERY_BLOCKED

2. 对冲孤儿或到期解绑失败
   -> ORPHAN_HEDGE_EMERGENCY

3. 已有退出/对冲订单在途
   -> 管理在途订单

4. 风险严重且期权退出可接受
   -> EXIT_PREFERRED

5. 风险严重、持续、期权退出更差
   -> HEDGE_READY

6. 止盈资格达到 80%
   -> TAKE_PROFIT_READY

7. 其余
   -> HOLD / WATCH
```

若已有退出活动开始：

- 禁止新增对冲敞口；
- 现有对冲只允许减仓；
- 对冲目标随卖方短腿剩余数量递减。

## 十、Deribit `BTC-PERPETUAL` 对冲落地

### 10.1 固定对冲工具

本版本对冲工具固定为：

```text
hedge_instrument = BTC-PERPETUAL
hedge_venue = DERIBIT
```

删除或停止使用：

- `adapter_preference = UNBOUND`
- Binance 对冲适配选择
- 运行时场所比较
- 其他期货/永续自动切换

选择理由已由用户确认：深度足够，并且与 Deribit 期权同场，便于统一账本、保证金、订单恢复与敞口管理。

### 10.2 对冲方向与目标

```text
SHORT_CALL_VERTICAL 风险上升
  -> BUY BTC-PERPETUAL

SHORT_PUT_VERTICAL 风险上升
  -> SELL BTC-PERPETUAL
```

对冲仍以压缩尾部为目标，不追求永久完全 Delta neutral。

对冲目标数量：

```text
target_hedge_qty
  = function(
      remaining_short_option_qty,
      current_structure_delta,
      hedge_reduction_ratio,
      BTC-PERPETUAL contract_size,
      min_trade_amount
    )
```

### 10.3 生命周期绑定

每条对冲记录必须带：

```text
parent_position_id
parent_session_id
parent_short_instrument
hedge_instrument = BTC-PERPETUAL
hedge_side
target_qty
actual_qty
remaining_short_qty
created_ts
last_reconciled_ts
```

硬不变量：

```text
remaining_short_option_qty == 0
  -> target_hedge_qty = 0

option_structure_state in {CLOSED, SETTLED}
  -> target_hedge_qty = 0
```

卖方短腿部分退出后，对冲目标必须立即按剩余短腿风险减少。卖方短腿完全归零后，对冲目标立即归零，不等待保护腿出售。

对冲依附的是剩余卖方期权风险，而不是保护腿残值。因此对冲可以在保护腿最终处置前归零，但绝不能在卖方期权风险已经消失后继续作为正常持仓保留。

### 10.4 对冲退出

所有对冲减仓与退出订单必须：

- 使用 Deribit `BTC-PERPETUAL`；
- 使用 `reduce_only=true`；
- 使用唯一 label；
- 先落 `PendingHedgeIntent`；
- 查询最终成交和实际持仓；
- 禁止因重复命令反向开仓。

在正常状态机中，不允许出现：

```text
期权卖方腿已退出或已结算
但 BTC-PERPETUAL 对冲仍被视为正常持仓
```

### 10.5 孤儿对冲故障态

交易所断线或订单失败可能造成技术性短暂未平。系统不能伪装不存在该风险，而必须进入：

```text
ORPHAN_HEDGE_EMERGENCY
```

处理规则：

- 禁止新增期权风险；
- 禁止创建新的对冲敞口；
- 持续发送 reduce-only 平仓意图；
- 每轮核对真实 `BTC-PERPETUAL` 持仓；
- 对冲实际数量归零前，不得将会话标记为 `CLOSED`；
- 必须记录从期权归零到对冲归零的持续时间。

目标不是声称分布式执行永远不会临时失配，而是保证任何失配都不能成为正常终态。

### 10.6 到期与交割保护

垂直价差两腿同到期。到期前必须运行结算保护：

```text
SETTLEMENT_GUARD
```

规则：

- 到期临近时禁止新增对冲；
- 根据剩余短腿实际敞口逐步将对冲目标归零；
- 期权进入 settlement/delivered 后立即核对真实持仓；
- 如果期权已经交割而对冲未归零，进入 `ORPHAN_HEDGE_EMERGENCY`；
- 对冲归零前不得归档会话。

## 十一、账本与持久化

### 11.1 最小账本

```text

exit_authorization_ledger
exit_campaign_ledger
hedge_ledger
pending_order_ledger
```

### 11.2 `_G()` 使用边界

执行实例内部可用 `_G()` 持久化：

- 当前推荐库摘要；
- 锁定计划；
- 硬交互处理结果；
- 当前软授权；
- 当前退出活动；
- 期权和对冲账本；
- pending intent；
- 已处理命令哈希。

不得用 `_G()`：

- 充当两个独立 FMZ 策略实例之间的可靠总线；
- 无限存储高频行情；
- 用本地账本覆盖交易所真实持仓事实。

### 11.3 启动恢复

启动顺序：

1. 读取交易所实际期权持仓。
2. 读取实际 `BTC-PERPETUAL` 持仓。
3. 读取活动订单和最终状态。
4. 读取本地账本与 pending intents。
5. 建立可解释映射。
6. 无法解释时进入 `RECOVERY_BLOCKED`。
7. 发现孤儿对冲时进入 `ORPHAN_HEDGE_EMERGENCY`。
8. 恢复完成前禁止开新仓。

## 十四、必须新增的关键测试

### 14.1 垂直价差唯一性

- 任意输入下推荐库都不能产生日历价差。
- 运行单文件不含日历方案生成入口。
- 两腿到期必须相同。

### 14.2 硬交互

- 无 `PLAN_EXECUTE` 时永不下单。
- 错误 `strategy_code` 拒绝。
- 错误 `quality_code` 拒绝。
- 错误 `plan_hash8` 拒绝。
- 推荐库刷新导致质量码变化时旧命令拒绝。
- 锁定计划不随推荐库刷新变化。

### 14.3 软授权

- 未授权时止盈达到 80% 只提示，不下单。
- 未授权时账本、行情、风险、对冲监控持续运行。
- 授权后条件未达到时不下单。
- 授权后条件达到时启动退出活动。
- 撤销授权后停止创建新退出订单，但继续管理已有订单。

### 14.4 低成本退出

- 无组合可执行报价时仍能按参考值进入止盈资格。
- 买回短腿订单不得突破 `max_total_exit_spend`。
- 多轮逐 tick 询价不阻塞主循环。
- 撤单后晚到成交能被发现。
- 部分短腿成交后保护腿不卖。
- 短腿完全归零后才允许卖保护腿。
- 保护腿无 bid 时进入 `LONG_RESIDUAL_ONLY`。

### 14.5 对冲生命周期

- 对冲工具只能是 `BTC-PERPETUAL`。
- 对冲记录必须有父持仓。
- 短腿部分平仓后对冲目标下降。
- 短腿归零后对冲目标归零。
- 对冲退出订单必须 `reduce_only=true`。
- 期权结算后存在对冲时进入 `ORPHAN_HEDGE_EMERGENCY`。
- 对冲未归零时会话不得 `CLOSED`。
