## 1. 本轮变更摘要

相对 v0.1/v0.2，本轮两点最大改动：

1. **取消 KPF 层与后续应用**（含 KPF-F 关键价位轴与 `kpf/` 项目内的 SLRP EV 因子）。结论：在当前验证范围内，KPF 与 SLRP 都没有作为「系统化选腿增量因子」的用武之地——**以 BS / IV surface / Greeks 为基础的市场期权定价，已充分覆盖 KPF/SLRP 试图补充的那部分空间风险**。继续保留 KPF 只会引入慢节奏批数据层、AWS 部署、陈旧/降级处理与「厚流动性层 = 不会突破的墙」式伪精确，却不改变任何一次选腿或对冲决策。
2. **并入对冲模块设计稿**。对冲从 v0.1 的「未建立、仅留接口」升级为有 v1.0 设计 + v0.3 代码 + 12 万样例模拟审计的正式第三模块。

由此，系统从 v0.1 的「信号 + KPF + 执行 + 对冲」四层，**收束为三模块主链**：

> **① 信号层（中性回路）** 决定何时、哪一侧 → **② 执行层（选腿+下单+记账）** 把方向落成可复核、可空跑、可记账的带保护腿期权结构 → **③ 对冲模块（持仓后）** 只在尾部恶化且期权退出更差时，给出期货腿压平意图。

三者通过版本化证据包协作，不互相硬连。系统必须保留拒绝交易能力：`NO_TRADE_BLOCKED`、`WAIT_CONFIRMATION`、执行不可行、期权退出困难、对冲条件不成立，都是正式输出而非失败。

## 2. 为什么取消 KPF / SLRP（取消的第一性论证）

不是「KPF 做错了」，而是**它要补的不确定性已被市场期权定价定价掉了**：

- **空间风险已在价里**。KPF 想回答「卖腿不利方向上，价格在威胁 short strike 前是否先撞到一层厚流动性」。但这层「难突破程度」正是市场用 IV surface 的斜率/凸度、各档 Greeks（尤其 delta=市场隐含触碰概率）定价出来的。执行层在 `SHORT_DELTA_RANGE=(0.15,0.45)` 内按 delta 选档时，已经在用市场对该 strike 触碰概率的报价；对冲模块的触界概率直接用 IV 做 BS 障碍模型。再叠一层 KPF 历史成交密度，是对同一信息的二次、且更滞后的重述。
- **耦合本就是可选覆盖层，删除是纯减法**（代码事实）：
  - 执行层选腿 `leg_selection.py` 的 `kpf_core / kpf_near / kpf_far` 三个软约束**默认 `None`**，缺省即纯 delta/腿宽逻辑；`PLAN_WEIGHTS["kpf"]=0.20` 只是综合分里一项权重。
  - 对冲 `hedge_risk.py` 把 KPF 仅当作三项**持续性确认之一**，且明确「KPF 不进入触界概率」。
  - 信号层（中性回路）**从不读取 KPF**——KPF 一直在下游。
- **节奏天然错配**。KPF 是读 Binance USD-M `aggTrades`、90 天历史/60 天新鲜窗的离线批处理；主链是 FMZ 实时 tick。强行接入要付 AWS 服务、`/latest` API、token+HTTPS+IP 白名单、陈旧/降级/重锚一整套复杂度，换来的是不改变决策的「装饰性空间美观」。
- **跨场基差也是净负担**。KPF 轴是 Binance 期货价系，执行在 Deribit 指数价系，本就需「按 Deribit 工作价重锚」的额外口径；删除后此口径一并消失。
- **12 万样例审计的旁证**。对冲模拟结论第 4 条已指出：KPF buffer 只能作持续性确认，一旦用它改动市场触界概率就会制造「虚假安心」。这正说明它不是独立信息源。

**SLRP 同理出局**：它是 `kpf/` 内基于成交结构的 EV 实验因子，与 KPF 同源、同样试图在市场定价之外再造一层空间/EV 估计，未通过「能改变某次具体决策」这一增量门槛。

> 取消后**不留空间盲区**：选腿的空间判断 = delta（市场隐含触碰概率）+ 腿宽 + S:PM 保证金释放；对冲的空间判断 = IV-based 触界概率 + GGR 负 Gamma 放大区（边界附近的非线性加速）。两者都来自实时市场，比 KPF 更及时、更少口径摩擦。

`kpf/` 项目本身作为资产保留在磁盘，但**移出本总纲的系统范围**，不再是任何运行层；其衍生工作线（KPF 服务器版封版与部署）一并取消。

## 3. 资产审计结论（三模块 · 第一性不变，只做减法）

### 3.1 ① 信号层 · 中性回路 v0.5.3 —— 时序 + 方向层

当前能力（代码事实）：

- 运行边界：`read_only_demo=True`，不选腿、不报价、不下单；`order_layer=external_execution_program`。
- 主链路（3 模块，进 `MODULE_SEQUENCE`）：`External Gate → Anchor → TMV-F`。
- 时序门：`NeutralRepair`（DIE+Anchor 状态机）判断修复窗口何时打开（其 UP/DOWN 是噪声、非方向）。
- 方向层（权威）：`EDB` 合成六类证据 —— TMV、CVD×价格、MACRO、FUNDING、SRD（25Δ 风险逆转）、GGR（全局 Gamma 区制空间钉），加权后验出 `edb_score / 置信 / 一致度 / 覆盖度`。
- 安全门：GGR 负 Gamma 放大区→砍/否决；宏观硬阻断；资金费率极端拥挤可 veto。
- 输出：`strategy_recommendation`（signal / strategy_code=put|call credit spread 侧 / strategy_type / 24h+48h 期号 / edb_lean / edb_support / edb_confidence），不含具体腿与报价。

**整合调整：零。** 信号层从不依赖 KPF；EDB 六证据里没有任何 KPF 项。第一性（只读时序+方向）完全不动。唯一需做的是**文档卫生**：`中性回路 - opus4.8\INTEGRATION_HANDOFF.md` 中把 KPF 列为「空间层」的三层叙事**已被本稿取代**，`PROJECT_MEMORY.md §16` 中「与下游 KPF 空间层整合」一句应改为「与执行层/对冲模块整合」。代码与封版状态不受影响。

方向正确性：✅ 正确。它只回答「窗口是否打开」「到期窗口哪一侧收敛」，把空间与执行全部外置——这正是取消 KPF 后系统要的形态。

### 3.2 ② 执行层 · Deribit v1.6.2 —— 选腿 + 下单 + 记账

当前能力（代码事实）：

- 两轮分离：`ROUND_MODE="PLAN"` 只生成方案库并排序；`ROUND_MODE="ORDER"` 按 `SELECTED_PLAN` 复核后才进下单轮。
- 安全门：`ALLOW_TRADING=False` 默认空跑；仅下单轮置 `True` 才真实开仓。
- 进场信号：只接受 `TRADE_SUPPORT_STRONG / WEAK`。
- 选腿宇宙：`SHORT_DTE_HOURS=(24,72)`、`SHORT_DELTA_RANGE=(0.15,0.45)`（卖权利金主驱动，delta 即市场隐含触碰概率）。
- 保护：`ENABLE_CALENDAR=False` 默认；同期垂直为默认候选，远期日历保护为可选能力（与旧设计稿强调日历不同，以代码为准）。
- 执行纪律：保护腿优先、maker-only、最多追一步、禁 market/taker、short 数量不超保护腿可用量、同向信号不加仓。
- 记账：保护库存、short 占用、状态机、启动对账、手续费/价差/保护成本/保证金释放（`ledger.py` / `accounting.py`）。

**整合调整（纯减法，第一性不变）：**

1. **删 `kpf_score` 评分项**。`PLAN_WEIGHTS` 由 `{win_rate:0.30, rr:0.30, kpf:0.20, signal:0.20}` 去掉 `kpf`，对剩余三项**等比重归一**得 `{win_rate:0.375, rr:0.375, signal:0.25}`（除以 0.80）。具体数值在回放阶段可微调，但 KPF 让出的 0.20 应主要回到 `win_rate/rr`——因为它们正是 IV/Greeks/真实报价驱动的项，承接了原 KPF 想补的空间角色。`plans.py`、`strategy.py`、单文件 `spm_calendar_protected_short_v1.py` 中 `weights["kpf"] * kpf_score` 一并移除。
2. **`leg_selection.py` 去掉 `kpf_core / kpf_near / kpf_far` 软约束**（本就默认 `None`）。`legsel_pick_nearest_delta` 退化为纯目标 delta 选档；`legsel_protection_candidates` 退化为纯腿宽区间 + 过度虚值过滤（`deep_otm_max_delta`）。
3. **空间口径改为期权链自洽**：short strike 由目标 delta 决定（避免贴现价、避免高触碰概率档），protection strike 由腿宽区间 + S:PM 保证金释放 + delta 下限决定。原「不要落进 KPF 争夺核心」由「按 delta 选档天然避开高触碰概率 strike」承接。
4. **设计稿对齐**：`deribit_spm_calendar_protected_short_v1_0_design.md` 中 §0/§3.1/§5.2/§6.3/§13 的 KPF 选腿/`kpf_context` 语言**降级为历史**，以本稿与 v1.6.2 代码为准。

第一性（两轮制 + `ALLOW_TRADING` 闸 + 保护腿优先 maker-only + S:PM 保证金释放 + 全损耗记账 + 同向不加仓）全部保留。方向正确性：✅ 正确，删 KPF 后更纯粹。

### 3.3 ③ 对冲模块 · 设计 v1.0 / 代码 v0.3 —— 持仓后尾部压平

第一性：期权腿已建立后，只回答三问——风险是否恶化、恶化是否持续、期权退出是否比期货腿更差；三者同时成立才生成**期货腿干跑对冲意图**。不入场、不判方向、不反手、不追求完全 delta neutral、第一版不真实下单。

收敛为五个决策变量：`touch_probability_now`（IV-based BS 障碍触界概率，缺 IV 用 delta 兜底并标低置信）、`touch_probability_drift`、`recent_deterioration_slope`（约 30 分钟）、`tail_exposure_acceleration`（净 delta/gamma 非线性加速）、`exit_vs_hedge_friction`。状态机：`NORMAL → WATCH → EXIT_PREFERRED → HEDGE_READY → HEDGE_ACTIVE`，外加 `MANUAL_REVIEW`。

模拟审计基准（`hedge_policy_simulation_summary.json`，seed=20260601，12 万样例，错配率 13.3%）：v0.3 决策脊柱方向正确，但需收紧——`HEDGE_READY` 必须更难触发，摩擦只在风险成立后决定「退出优先还是对冲优先」、不得提前改变风险状态。

**整合调整（纯减法 + 阈值重标定，第一性不变）：**

1. **持续性确认从三项降为两项**：去掉「KPF Buffer Broken」（`_kpf_buffer_adverse`、`_BAD_KPF_BUFFER_STATES`、`kpf_context` 入参、`entry_kpf_buffer_state` 锚字段全部移除）。持续性 = `{EDB_ADVERSE, GGR_ADVERSE}`，其中 **EDB 是唯一方向证据入口**，GGR 是负 Gamma 非线性加速的例外修正。
2. **持续性分级重标定**：`0 项→LOW / 1 项→MEDIUM / 2 项→HIGH`。`HEDGE_READY` 闸沿用 v1.0：`hard_risk 且 persistence ≥ MEDIUM 且 hedge_friction_advantage`——即至少 EDB 或 GGR 一项不利 + 严格 hard_risk + 明确摩擦优势，三者同时才放行，保持「`HEDGE_READY` 最难触发」。
3. **采纳 v1.0 收紧项**（覆盖 v0.3 代码当前阈值）：`WATCH` 设为「风险上升但尚不可对冲」独立通道，避免无持续性的 drift 直接变对冲意图；`EXIT_PREFERRED` 仅在风险已上升且期权退出仍可接受时出现；hard boundary 击穿 / 极高 `p_touch` 在 persistence≥MEDIUM 且摩擦占优时走直达 `HEDGE_READY`。
4. **尺寸**沿用 LIGHT/MEDIUM/FULL 三档封顶比例，不追求完全中性。
5. **触界概率公式不变**——它本就是 IV-based，删 KPF 不触碰它（反而消除了「用 KPF 改触界概率」的虚假安心风险，与审计结论第 4 条一致）。

第一性（持仓后尾部压平、EDB 唯一方向入口、GGR 仅非线性修正、`DRY_INTENT_ONLY`、独立对冲开关）全部保留。方向正确性：✅ 正确。

> 注：v1.0 设计稿 §6.3「KPF Buffer Broken」与 §4/§11 中 `entry_kpf_buffer_state`、`kpf_context` 字段，以及代码 v0.3 的对应实现，按本节降级。落地时把 v0.3 代码升到「持续性两项 + v1.0 收紧状态机」即得对冲模块的整合目标版本（建议记为 v0.4 / position_risk.v0.4）。

## 4. 目标架构（取消 KPF 后的数据通路）

```text
Public Market Data
  → ① 信号 FMZ（中性回路 v0.5.3，只读）
       → SignalEvidencePackage         （时序 + 方向 + EDB + GGR）

SignalEvidencePackage + operator limits + Deribit 实时期权链(IV surface/Greeks)
  → ② 执行 FMZ · PLAN
       → ExecutionPlanPackage          （delta/DTE/RR/S:PM 驱动方案库；无 kpf_score）

Approved ExecutionPlanPackage
  → ② 执行 FMZ · ORDER
       → PositionLedgerPackage + EntryRiskAnchor   （成交即落不可覆盖入场锚）

PositionLedgerPackage + 实时行情(IV/Greeks/退出摩擦) + SignalEvidencePackage(EDB/GGR)
  → ③ 对冲模块（驻执行 FMZ 内，读账本同进程）
       → PositionRiskPackage / HedgeIntentPackage  （DRY_INTENT_ONLY）
```

关键原则：

- 上游只给证据，不给强制下单命令。
- 执行层只接受明确 `TRADE_SUPPORT_*`，并在 PLAN 轮重新验证腿、报价、S:PM、执行摩擦。
- **全系统空间/定价口径统一来自 Deribit 实时期权链**：无外置批数据、无 90 天历史成交重算、无跨场基差重锚。
- 持仓后模块只做风险压缩，不重判入场方向。
- 每个包都有 `schema_version`、生成时间、输入来源、数据质量、过期时间（`ttl_ms`/`expires_at_ms`）、拒绝原因。

| 模块 | 应负责 | 不应负责 |
| --- | --- | --- |
| ① 中性回路 | DIE+Anchor 修复时序门；EDB 合成到期窗口方向；输出 `TRADE_SUPPORT_*` 或拒绝态 | 不选腿；不报价；不下单；不做持仓后对冲 |

| ③ 对冲模块 | 持仓后监控触界恶化、持续性、退出摩擦；必要时给期货腿干跑对冲意图 | 不作入场；不判方向；不反手；不自动改期权账本；第一版不真实下单 |

## 6. 契约（KpfSpatialPackage 已删除）

四个版本化包是模块间唯一合法耦合面（即便同进程，执行层也只看 `SignalEvidencePackage`，不读 EDB 内部状态）。本轮删除 `KpfSpatialPackage`，新增/锐化对冲两包。

  "schema_version": "nrd.integration.signal.v0.1",
  "source_demo_version": "0.5.3",
  "generated_at_ms": 0, "ttl_ms": 300000,
  "asset": "BTC", "current_price": 0,
  "support_label": "TRADE_SUPPORT_WEAK",
  "direction_bias": "SHORT_CALL",
  "confidence": 0,
  "edb": {"edb_score": 0, "lean": "", "agreement": 0, "coverage": 0, "ggr_gate": {}},

## 7. 主流程

### 7.1 日常准备
1. 启动 ① 中性回路只读观察，记录 `SignalEvidencePackage`。
2. 信号非 `TRADE_SUPPORT_*` → 停在观察，不进期权计划。
3. 确认 ② 执行 FMZ 可读到 Deribit 期权链（IV/Greeks/盘口）与最新信号包（`ttl` 内）。

### 7.2 建仓前
1. ① 给窗口：`NeutralRepair.is_active=true`、EDB `support_label=STRONG/WEAK`、GGR/macro/funding 未 veto。
2. 生成执行层配置：`SIGNAL_STATE / DIRECTION_BIAS / SIGNAL_CONFIDENCE`（无任何 KPF 参数）。
3. ② 跑 `ROUND_MODE=PLAN`：在 DTE/Delta 硬范围内枚举候选，按 `win_rate/rr/signal` + S:PM 保证金释放打分排序，输出方案库与拒绝漏斗。
4. 人工选方案 → 才允许 `ROUND_MODE=ORDER`；`ALLOW_TRADING=True` 是最后一步人工确认。

### 7.3 建仓中
1. 保护腿优先；每腿 maker-only、最多追一步。
2. 保护腿未成交不卖 short；short 未成交按代码配置尝试卖回保护腿避免裸保护。
3. 成交写账本 + 落 `EntryRiskAnchor`，生成 `PositionLedgerPackage`。

### 7.4 持仓后
1. 同向信号不加仓，只记录确认。
2. 信号转 `NO_TRADE_BLOCKED / AMBIGUOUS` → 进退出/复核。
3. ③ 对冲模块读账本 + 实时行情(IV/Greeks/退出摩擦) + EDB/GGR：算五变量 → 定状态。
4. 期权退出可行 → 优先期权退出/减风险（`EXIT_PREFERRED`）。
5. 风险硬成立 + 持续性 ≥ MEDIUM + 期货摩擦占优 → `HEDGE_READY`，出 `HedgeIntentPackage`（干跑）。
6. 期货腿建立/调整/解除必须独立记账，不覆盖原期权账本。

## 8. 拓扑与部署（取消 KPF 后大幅简化）

### 8.1 拓扑：模块化单体，运行时分离，契约即边界
- **两个 FMZ 策略，一台托管者**：① 信号层（约 8000 行、只读、发信号包）与 ② 执行层（持私钥、PLAN/ORDER 下单）物理隔离，缩小爆炸半径。③ 对冲模块 v0.x 为 `DRY_INTENT_ONLY`，**驻 ② 执行策略内**（同进程直读期权账本与实时行情，省去把整本账本跨总线序列化）；契约边界（`PositionRiskPackage`）保留，待 Phase 3 `ALLOW_HEDGE_TRADING` 上线、尤其若对冲改去 Binance 永续另持密钥时，可独立成第三策略。
- 即便同进程，模块间也只经版本化包耦合，随时能再拆、能回放、能审计。

### 8.2 包总线（不再需要公网）
- 取消 KPF 后，**唯一跨策略包是 `SignalEvidencePackage`**（① 写、② 读），双方同在一台 FMZ 托管者 → **优先本地通路**：总线绑 `127.0.0.1`，或托管者共享路径上的原子写文件（write-tmp + rename）。**触发交易的信号数据不出本机，无需公网暴露、无需 token+HTTPS+IP 白名单那一整套**（那套原本是为 KPF 远程重数据准备的，现已不需要）。
- **待验证 spike**：同托管者两机器人是否共享 loopback 与可写路径（取决于托管者是否对每个机器人单独容器化）。本地不通则回退一个最小远程总线（同 token 方案），但**仅传信号包**，规格远小于原 KPF 服务。
- **读写分权**：① 只持 `/signal` 写 token；② 只持 `/signal` 只读 token。
- **fail-safe**：总线不可达 = 拿不到新信号 = 不开新仓（已有持仓由 ② 本地账本管）；配合包 `ttl_ms` 自动拒绝过期包。
- **审计/回放**：每包带时间戳落盘 + 账本记录「基于哪份包下单」。

### 8.3 部署对比（v0.2 → v0.3）
| 项 | v0.2（含 KPF） | v0.3（取消 KPF） |
| --- | --- | --- |
| 外置服务 | AWS KPF 服务（2 vCPU/4–8GB、数十 GB 盘、日级 cron、90 天滚动） | **无** |
| 公网通路 | KPF `/latest` token+HTTPS+IP 白名单 | **无（信号包走本机 loopback/共享文件）** |
| 陈旧/降级 | KPF last-good/degraded/重锚一整套 | **无** |
| 跨场基差 | Binance↔Deribit 重锚口径 | **无（统一 Deribit 价系）** |
| 运行策略数 | 2 FMZ + 1 AWS 服务 | **2 FMZ（对冲驻执行内）** |

## 9. 上线门槛（三道开关 + 两道闸）

三个独立安全开关，互不连带：
- `ALLOW_TRADING`（期权下单，默认 False）。
- `ALLOW_HEDGE_TRADING`（期货腿对冲，默认 False，Phase 3 才引入，与 `ALLOW_TRADING` 分离）。
- 全局 kill-switch（一键全停）。

两道闸（针对期权执行）：
- **闸 A · 管道验证（校准前可做）**：一次性、手动、定额极小、全程盯守的真实下单循环（保护腿→短腿→成交→记账→平仓）。临时只对这一单开 `ALLOW_TRADING=True`，跑完立即关。测的是水管物理可行，不是策略 edge。
- **闸 B · 信号驱动实盘（必须校准后）**：硬前提 = 信号层 P0 阈值已按 `demo/CALIBRATION_PLAN_V0.5.md` 校准 + `nr_threshold_profile` 由 `relaxed_test` 换 production + 全链空跑拒绝链一致。

**校准完成前，除闸 A 外全链 `ALLOW_TRADING=False`；对冲始终 `DRY_INTENT_ONLY` 直到独立回放通过。**

## 11. 不变量（整合必须守）

1. 信号层、执行层、对冲层最终可在 FMZ 内合并，但合并的是轻量运行逻辑；不引入任何外置重数据计算。
2. 方向只来自 `SignalEvidencePackage` 或人工明确配置；执行层与对冲层都不自行推断方向。
3. 执行才是唯一期权下单点：不裸卖、保护腿在卖方腿外侧、固定风险比例、硬门控不因错过行情放宽。
4. GGR 否决 / 宏观·资金硬阻断 / DIE+Anchor 窗口未开 → 执行层不得越过中性回路的「不可交易」结论。
5. 对冲模块不自动改期权账本，只追加独立 hedge 账本；`HEDGE_READY` 必须同时满足风险、持续性、摩擦优势。
6. schema 边界不变：中性回路仍 `nrd.schema.v0.5.3`，整合通过消费其输出实现，不往其 payload 塞执行字段。
7. 所有自动化前先有可回放输入包/输出包，且能被人工打开复核。

## 15. 一句话总纲

中性回路总系统是「证据驱动的分段式卖方风险压缩流程」：**① 中性回路只确认窗口与方向证据；② Deribit 执行层只把被放行的方向，在 DTE/Delta 硬范围内、以期权链自身的 IV surface/Greeks 与 S:PM 保证金释放为口径，转成可复核、可空跑、可记账的带保护腿结构；③ 对冲模块只在持仓后、当触界恶化且有 EDB/GGR 持续性、且期权退出比期货更差时，给出干跑期货腿压尾意图。** 三者通过版本化证据包协作；空间与定价口径统一来自实时市场，故取消 KPF/SLRP 后系统更轻、更快、更少口径摩擦，且不留空间盲区。
