# 中性回路整合思路方案参考稿 v0.1

日期：2026-06-02

定位：总纲级整合思路参考稿。本稿用于比较不同整合路径，并给出在当前 FMZ/Deribit/本地模块条件下更正确、通顺、具备 polish 感的执行链路建议。本文不改动现有代码，不授权实盘，不把尚未标定的模块伪装成生产能力。

---

## 0. 一句话结论

推荐路线是：

```text
信号层保持独立只读
  -> 通过版本化 SignalEvidencePackage 给执行层
执行层升级为统一 ExecutionSession
  -> VRP 作为 PLAN 内的建仓前定价硬门
  -> PLAN/ORDER 不再靠重启切换，而是靠同一会话内的 approval intent 状态机
  -> 记账、持仓管理、对冲监控并入执行层的 Manage/Hedge 段
```

VRP 应放入执行层，不放入信号层。它不是方向因子，而是“被信号层允许的窗口/卖出侧，现在 Deribit 市场定价是否足够补偿风险”的执行前定价门。

对冲模块后续应并入执行层，但身份是持仓后风险压缩器，不是入场器，也不是方向模型。

## 1. 当前整合进度盘点

| 模块 | 当前状态 | 主要事实 | 整合判断 |
| --- | --- | --- | --- |
| 信号层 | 已有 v0.5.4 外部交付物 | 只读、`signal_and_strategy_only`、输出信号/方向/期号/策略建议 | 保持独立，向执行层发包，不下沉到选腿/下单 |
| Deribit 执行层 | v1.6.2 | PLAN/ORDER 两轮、方案库、S:PM、maker-only、账本、持仓状态机 | 保留安全门，但重构体验形态 |
| 对冲模块 | 执行仓库已有纯函数/干跑输出 | `PositionRiskPackage` / `HedgeIntentPackage`，不下单、不改账本 | 并入执行层 Manage/Hedge 段 |
| VRP | v1.1 阶段性封版，仍独立 | window gate + candidate gate；已写执行层收口契约；未嵌入执行层 | 进入执行层 PLAN，且只读硬门 |
| KPF/SLRP | 已从新总纲路径取消 | 不再作为当前系统主链路依赖 | 不回接，不作为方向或空间硬门 |

当前最不通顺的体验点不是单个模块错，而是执行链被人为拆成两段：

```text
ROUND_MODE=PLAN
  -> 看方案库
  -> 人工记方案号
  -> 改 SELECTED_PLAN / ALLOW_TRADING
  -> 重启 ROUND_MODE=ORDER
  -> 再进入下单/记账/管理
```

这个结构适合 demo 安全验证，但不适合后续长期运行。它把“审计安全”误做成了“运行割裂”。

## 2. 模块第一性边界

### 2.1 信号层第一性

信号层只回答：

```text
当前是否存在中性回路修复窗口？
方向证据是否收敛？
给 24h/48h 到期窗口怎样的策略侧建议？
```

信号层不负责：

- 不读取执行层账本。
- 不判断具体腿。
- 不计算权利金是否足够贵。
- 不判断保证金释放。
- 不生成订单。
- 不管理持仓后对冲。

### 2.2 VRP 第一性

VRP 只回答：

```text
被信号层放行的 expiry/side，在当前 Deribit IV/RV/盘口/保护腿成本下，是否值得卖波动率？
```

VRP 不负责：

- 不判断方向。
- 不选择期号。
- 不进入主排序权重。
- 不替代 S:PM。
- 不替代 maker-only 执行纪律。
- 不覆盖 `NO_TRADE_BLOCKED`。
- 不因为“权利金很肥”降低 EDB/GGR/宏观门槛。

因此 VRP 的正确落点是执行层 PLAN 内，不是信号层。

### 2.3 执行层第一性

执行层只回答：

```text
在被允许的方向和窗口里，当前是否存在可复核、可记账、可控摩擦、可执行的期权结构？
如果用户授权，是否能按 maker-only 和账本纪律执行？
```

执行层不负责：

- 不重新发明方向判断。
- 不绕过信号/VRP/GGR/安全门。
- 不把风控对冲变成加仓逻辑。

### 2.4 对冲模块第一性

对冲模块只回答：

```text
持仓后风险是否相对入场锚点恶化？
恶化是否有持续证据？
期权退出是否比期货临时对冲更差？
```

对冲模块不负责：

- 不决定入场。
- 不重做 EDB。
- 不重做 VRP。
- 不把期货腿变成方向交易。
- 第一阶段不自动下单，只输出干跑意图。

### 2.5 最终两层落地分层

若最终只能保留两层，建议命名为：

```text
第一层：Signal / Evidence Layer（信号与前置证据层）
第二层：Execution / Risk Layer（执行、账本、组合风险与持仓管理层）
```

关键原则：

```text
信号层只产生“是否值得进入执行评估”的前置信息；
执行层决定“能否形成可执行结构、是否授权执行、如何管理风险和退出”。
```

#### 应放入信号层的内容

信号层可以并列输出这些前置信息：

| 类别 | 内容 | 输出形态 | 边界 |
| --- | --- | --- | --- |
| 时序窗口 | DIE + Anchor 修复窗口、窗口状态、TTL | `timing_window` | 不下单，不选腿 |
| 方向证据 | EDB lean、support、confidence、coverage、reason codes | `direction_evidence` | 不重写成仓位大小 |
| 到期建议 | 24h/48h 可观察 expiry、策略侧建议 | `strategy_recommendation` | 只建议窗口，不选具体 strike |
| GGR/宏观/资金费率摘要 | regime、veto、trend/vol expansion warning、macro/funding pressure | `pre_trade_context` | 作为前置信息，不做执行价格判断 |
| 数据质量 | source freshness、missing fields、calibration profile、signal package version | `data_quality` | 数据差可以阻断信号发包 |
| 禁止/等待状态 | `NO_TRADE_BLOCKED`、`WAIT_CONFIRMATION`、原因 | `reject_state` | 阻断态不得被执行层 VRP 肥度覆盖 |

信号层不应放：

- VRP pricing gate。
- Deribit 费率、价差、滑点、maker 成交判断。
- S:PM 保证金模拟。
- 具体 short/protection leg。
- `ApprovalIntentPackage`。
- 账本、持仓、对冲、止盈、滚动。
- Kelly/sizing、组合 vega/gamma budget。
- 交易后收益归因。

原因：这些都依赖执行账户、Deribit 当前盘口、账本状态、持仓状态或成交事实。放进信号层会把只读前置模型污染成半执行系统。

#### 应放入执行/风险层的内容

执行层应承接所有“从前置信息到真实交易事实”的内容：

| 类别 | 内容 | 输出形态 | 边界 |
| --- | --- | --- | --- |
| 信号接收 | 读取并校验 `SignalEvidencePackage` | `accepted_signal` | 不重判信号内部逻辑 |
| VRP 定价门 | window gate、candidate gate、hurdle、edge、reason codes | `VrpGatePackage` | 只做硬过滤，不进方向判断 |
| 候选结构 | short/protection leg 枚举、DTE/delta/strike、报价 | `ExecutionPlanPackage.menu` | 只在信号允许侧枚举 |
| 成本与保证金 | Deribit fee、价差/滑点、S:PM、full-burn、net credit | `cost_report` / `spm_report` | 复用执行层 canonical 原语 |
| 组合硬预算 | 并发持仓、vega/gamma、保证金占用、回撤熔断 | `PortfolioRiskBudgetPackage` | 第一版保守硬门，不做伪精确 Kelly |
| 授权体验 | plan lock、plan hash、TTL、armed preview、撤销/过期 | `ApprovalIntentPackage` | 解决 PLAN/ORDER 重启割裂 |
| 下单执行 | maker-only、保护腿优先、只追一步、kill switch | `OrderIntentPackage` / fills | 只有显式开关才真实下单 |
| 账本 | protection inventory、short ledger、reconcile、entry anchor | `PositionLedgerPackage` | 所有持仓事实以账本为准 |
| 持仓风险 | touch probability drift、tail acceleration、exit friction | `PositionRiskPackage` | 只在持仓后运行 |
| 对冲 | hedge intent、ALLOW_HEDGE_TRADING、adapter preference | `HedgeIntentPackage` | 默认干跑，不作为方向交易 |
| 赢家管理 | 止盈、时间退出、末日 gamma 退出、roll dry-run | `PositionManageDecision` | 复杂自动 roll 后置 |
| 归因 | theta、方向、IV/RV、费用、滑点、保护腿、hedge、residual | `AttributionPackage` | 最小归因应随账本同步 |
| 全链回放 | signal -> plan -> cost -> net P&L | `ReplayExpectationPackage` | 小额实盘前硬前提 |

#### 两层最终数据流

```text
Signal / Evidence Layer
  -> SignalEvidencePackage
      - timing_window
      - direction_evidence
      - strategy_recommendation
      - pre_trade_context
      - data_quality
      - reject_state

Execution / Risk Layer
  -> accept signal
  -> VRP_PRICE_GATE
  -> PLAN_READY
  -> PORTFOLIO_BUDGET_CHECK
  -> APPROVAL_INTENT
  -> ARMED_PREVIEW
  -> COMMIT_ORDER
  -> POSITION_LEDGER
  -> POSITION_MANAGE
  -> HEDGE_WATCH
  -> ATTRIBUTION / REPLAY
```

#### 两层合并后的命名口径

不要再单独叫“风险层”作为第三层；风险能力并入执行层，但在执行层内部保持子域：

```text
Execution / Risk Layer
  - Pricing Gate Domain: VRP
  - Plan Domain: leg selection / S:PM / cost
  - Approval Domain: plan lock / armed preview
  - Order Domain: maker-only execution
  - Ledger Domain: accounting / inventory / reconcile
  - Portfolio Risk Domain: sizing budget / drawdown fuse
  - Position Risk Domain: hedge watch / winner management
  - Attribution Domain: P&L source decomposition
```

这样最终只有两层，但不会丢失风险模块的第一性。

## 3. 整合方案候选

### 方案 A：保守桥接型

形态：

```text
信号层独立运行
人工读取信号包
执行层继续 PLAN/ORDER 两轮
VRP 先独立跑，只给人工参考
对冲模块独立干跑
```

优点：

- 最安全，改动最少。
- 不会因接口错误导致自动交易。
- 适合短期继续 demo/观察。

缺点：

- 用户体验最差。
- 信号、VRP、执行、对冲证据链分散。
- 容易复制旧信号、旧方案号、旧 KPF/旧报价。
- 难以形成端到端回放。

适用阶段：只适合继续做 1-2 轮验证，不适合作为下一阶段主形态。

### 方案 B：执行层内聚型

形态：

```text
信号层独立
执行层成为统一 ExecutionSession
VRP 进入执行层 PLAN
PLAN/ORDER 合并为同一运行内状态机
记账 + 对冲监控并入执行层
```

核心体验：

```text
PLAN_READY
  -> 用户锁定方案
  -> APPROVAL_INTENT
  -> 执行前重价/重验
  -> ARMED_PREVIEW
  -> 显式授权后 COMMIT_ORDER
  -> POSITION_MANAGE
  -> HEDGE_WATCH
```

优点：

- 最大程度保留模块第一性。
- 去掉重启和手填方案号的割裂感。
- VRP 与 S:PM、记账、EntryRiskAnchor 共享执行层 canonical 原语。
- 对冲自然读取执行层账本与入场锚点。
- 能形成端到端审计包。

缺点：

- 需要改执行层状态机和面板。
- 需要新增 approval intent 持久化对象。
- 需要给 FMZ 运行环境做一次交互/持久化能力验证。

适用阶段：推荐作为主线。

### 方案 C：全链编排器型

形态：

```text
信号层、执行层、VRP、对冲层都由一个外部 Orchestrator 调度
各模块以服务/API 形式存在
Orchestrator 统一管理会话、授权、回放、审计
```

优点：

- 架构最优雅，最接近长期平台。
- 回放、审计、权限、UI 可以做得最好。
- 可以支持多标的、多账户、多策略。

缺点：

- 当前落地成本最高。
- 需要额外服务、部署、鉴权、运维。
- 对 FMZ 托管者网络/文件隔离有更多假设。
- 当前策略还在小额/干跑校准期，过早平台化会掩盖核心策略风险。

适用阶段：可作为中长期方向，不建议现在主推。

### 方案 D：单体 FMZ 策略型

形态：

```text
信号层 + 执行层 + VRP + 对冲全部合成一个 FMZ 单文件
同一进程完成观察、计划、授权、下单和管理
```

优点：

- 数据互联最直接。
- 不需要跨机器人通信。
- 部署数量少。

缺点：

- 只读观察与持钥下单混在一起，爆炸半径变大。
- 单文件长度和状态复杂度迅速失控。
- 信号层开发/回放会被执行层安全门牵制。
- 不利于保持信号层第一性。

适用阶段：不推荐作为主线。仅可在极简本地模拟中临时使用。

## 4. 推荐方案：执行层内聚型 + 信号层独立

推荐拓扑：

```text
Signal FMZ Robot
  -> SignalEvidencePackage
  -> local bus / shared file / fallback HTTP

Execution FMZ Robot
  -> ExecutionSession
      -> PRICE_GATE(VRP)
      -> PLAN_READY
      -> APPROVAL_INTENT
      -> ARMED_PREVIEW
      -> COMMIT_ORDER
      -> POSITION_MANAGE
      -> HEDGE_WATCH
```

### 4.1 新的主状态机

```text
IDLE
  -> SIGNAL_OBSERVED
  -> PRICE_GATE
  -> PLAN_READY
  -> PLAN_LOCKED
  -> APPROVAL_INTENT
  -> ARMED_PREVIEW
  -> ORDER_COMMITTING
  -> POSITION_OPEN
  -> POSITION_MANAGE
  -> HEDGE_WATCH
  -> EXIT_OR_REUSE
  -> CLOSED
```

核心变化：

- `ROUND_MODE` 不再是用户体验的主开关。
- `SELECTED_PLAN` 不再要求重启手填。
- 方案编号升级为方案身份的一部分，而不是唯一安全锚点。
- 真正的安全锚点是 `ApprovalIntentPackage`。

### 4.2 ApprovalIntentPackage

建议新增：

```json
{
  "schema_name": "ApprovalIntentPackage",
  "schema_version": "nrd.integration.approval_intent.v0.1",
  "session_id": "",
  "signal_package_id": "",
  "plan_id": 0,
  "plan_hash": "",
  "plan_created_ts": 0,
  "approval_created_ts": 0,
  "ttl_sec": 90,
  "approval_state": "LOCKED|ARMED|EXPIRED|CANCELLED|CONSUMED",
  "allow_real_order": false,
  "operator_note": "",
  "precommit_checks": {
    "signal_fresh": false,
    "vrp_rechecked": false,
    "spm_rechecked": false,
    "quotes_rechecked": false,
    "ledger_rechecked": false,
    "spread_ok": false,
    "maker_only": true
  }
}
```

解释：

- `plan_id` 仍存在，便于人看。
- `plan_hash` 防止方案重排、刷新或同号碰撞。
- `ttl_sec` 防止旧报价进入执行。
- `approval_state` 让同一运行内可撤销、可过期、可消费。
- `allow_real_order` 仍需显式打开，不因锁定方案自动下单。

### 4.3 ExecutionSessionPackage

建议执行层每轮面板和日志都围绕一个会话包：

```json
{
  "schema_name": "ExecutionSessionPackage",
  "schema_version": "nrd.integration.execution_session.v0.1",
  "session_id": "",
  "state": "PLAN_READY",
  "signal": {},
  "vrp_gate": {},
  "plan_menu": [],
  "locked_plan": {},
  "approval_intent": {},
  "order_intent": {},
  "position_ledger": {},
  "position_risk": {},
  "reason_codes": []
}
```

好处：

- UI 显示的不是一堆孤立表，而是一条执行故事。
- 回放时按 `session_id` 串起信号、VRP、计划、授权、下单、账本、对冲。
- 模块可分开写，但用户只看到一个会话。

### 4.4 面板 polish 目标

执行层面板改成三段 cockpit：

```text
1. 信号与资格
   signal_state / direction / freshness / reject reason

2. 建仓计划
   VRP gate / rejected funnel / S:PM / plan menu / locked plan / cost

3. 持仓管理
   ledger / protection inventory / short status / position risk / hedge intent
```

视觉/信息原则：

- 不再用“计划轮/下单轮”作为用户主语。
- 用“当前会话状态”作为主语。
- 只显示下一步可做动作，不显示无关操作提示。
- 每个拒绝都要落到一个明确门：信号、VRP、报价、S:PM、库存、授权、账本、对冲。
- `VRP edge` 与 `plan_ev` 并列时必须标注口径差异，不相加。

## 5. VRP 的最终位置

VRP 放执行层，理由：

1. VRP 使用 Deribit 实时期权链、盘口、IV、Greeks、RV 和完整摩擦，这些都是执行层资源。
2. VRP 判断的是“价格够不够补偿”，不是“方向对不对”。
3. VRP 需要复用执行层 `accounting`、`plans`、`hedge_risk` 中的 canonical 原语，避免两套费率/IV 真值源。
4. VRP 的输出要写入候选对象和 `EntryRiskAnchor`，天然贴近执行层。
5. 若放入信号层，会污染信号层第一性，使信号层从“只读证据层”变成半执行层。

推荐接口：

```text
SignalEvidencePackage 放行
  -> 执行层按 side/expiry 跑 VRP window gate
  -> PASS 窗口枚举候选
  -> candidate gate 筛结构
  -> 双门 PASS 才进入 S:PM / PLAN 排序
```

## 6. 对冲模块并入路径

对冲模块不应该作为独立用户入口长期存在。它应变成执行层持仓管理的一段：

```text
POSITION_OPEN
  -> 读取 EntryRiskAnchor
  -> 读取当前 price / dte / iv / delta / gamma
  -> 读取 EDB aggregate / GGR / KPF buffer 或其替代边界事实
  -> 生成 PositionRiskPackage
  -> 若 HEDGE_READY，生成 HedgeIntentPackage
```

开关原则：

- `ALLOW_TRADING` 只管期权开仓。
- `ALLOW_HEDGE_TRADING` 单独存在，默认 false。
- 第一阶段 `HedgeIntentPackage.execution_mode = DRY_INTENT_ONLY`。
- 对冲 adapter 只执行 `HedgeIntent`，不参与风险判断。

对冲模块的 polish 显示：

```text
风险状态: NORMAL / WATCH / EXIT_PREFERRED / HEDGE_READY / MANUAL_REVIEW
触界概率: now / entry / drift
持续性: EDB / GGR / buffer confirmations
动作建议: exit preferred / wait / dry hedge intent
```

## 7. 落地环境评估

| 因素 | 当前约束 | 对方案的影响 |
| --- | --- | --- |
| FMZ 单文件交付 | 执行层可构建单文件；信号层也偏单文件 | 不宜合并成超大单体，优先两个机器人互联 |
| 私钥安全 | 执行层持钥，信号层只读 | 信号层和执行层物理隔离更合理 |
| 模块体量 | 信号层约大文件，执行层也较复杂 | 单体化会降低可维护性 |
| VRP 状态 | v1.1 阶段性封版，但 edge 未生产级证明 | 可进 PLAN 只读硬门，不可解实盘 |
| 对冲状态 | 纯函数/干跑可用 | 可进管理面板，不宜自动下单 |
| 用户体验诉求 | 不接受重启式 PLAN/ORDER | 必须引入同一会话 approval intent |
| 回放审计 | 后续需要端到端证据链 | 必须以 session_id 串包 |
| 当前策略阶段 | 小额/干跑/校准阶段 | 不能走全自动流水线 |

## 8. 方案评估矩阵

评分：5 最优，1 最差。

| 方案 | 模块第一性 | 正确性 | polish 体验 | 安全性 | 落地成本 | 回放审计 | 当前推荐 |
| --- | ---: | ---: | ---: | ---: | ---: | ---: | --- |
| A 保守桥接 | 5 | 4 | 1 | 5 | 5 | 2 | 短期过渡 |
| B 执行层内聚 | 5 | 5 | 5 | 4 | 3 | 5 | 主推荐 |
| C 全链编排器 | 5 | 5 | 5 | 4 | 1 | 5 | 中长期 |
| D 单体 FMZ | 2 | 3 | 4 | 2 | 3 | 3 | 不推荐 |

综合判断：

```text
当前最优 = 方案 B
长期演进 = 方案 B -> 方案 C
短期兜底 = 方案 A
避免路线 = 方案 D
```

## 9. 推荐落地阶段

### Phase 0：文档定稿与接口冻结

目标：

- 冻结 `ExecutionSessionPackage`。
- 冻结 `ApprovalIntentPackage`。
- 冻结 VRP 进入执行层 PLAN 的最小字段。
- 冻结对冲并入执行层 Manage/Hedge 的输入输出。

验收：

- 所有模块第一性边界写清楚。
- 不再出现“VRP 进信号层”的歧义。
- 不再出现“对冲模块可入场”的歧义。

### Phase 1：体验层重构，先不接 VRP 下单

目标：

- 执行层保留现有方案库生成。
- 新增同一运行内 `PLAN_LOCKED` / `APPROVAL_INTENT` / `ARMED_PREVIEW`。
- 替代重启式 `ROUND_MODE=ORDER + SELECTED_PLAN` 的用户流程。
- 仍然 `ALLOW_TRADING=False`，只展示将执行订单。

验收：

- 用户可以在同一运行会话里锁定方案。
- 锁定方案有 `plan_hash` 和 TTL。
- TTL 过期必须退回 PLAN_READY。
- 方案刷新后 hash 不一致不能执行。
- 面板不再提示“重启进入下单轮”作为主流程。

### Phase 2：VRP 只读嵌入 PLAN

目标：

- 按 v1.1 收口契约把 VRP 绑定执行层 canonical 原语。
- 只跑 EDB 背书侧。
- 加 window gate / candidate gate 漏斗。
- 输出 VRP 字段进候选和面板。

验收：

- VRP BLOCK 的候选不进入最终可锁定方案。
- VRP 不进主排序权重。
- `ALLOW_TRADING=False` 下全链空跑不产订单。
- tests 覆盖 window/candidate/data downgrade/EntryRiskAnchor 映射。

### Phase 3：EntryRiskAnchor 与账本增强

目标：

- 成交后把 VRP 血缘写入 `EntryRiskAnchor`。
- 账本记录 `signal_package_id`、`plan_hash`、`approval_id`。
- 形成完整 session replay 线索。

验收：

- 每笔 short 都有不可覆盖的入场锚点。
- 锚点能追溯信号、VRP、计划、授权、报价和 S:PM。
- 对冲模块只读锚点，不反向重做 VRP。

### Phase 4：对冲并入 Manage/Hedge

目标：

- 面板加入 `PositionRiskPackage`。
- `HEDGE_READY` 时展示 `HedgeIntentPackage`。
- 默认 `DRY_INTENT_ONLY`。
- `ALLOW_HEDGE_TRADING` 独立于 `ALLOW_TRADING`。

验收：

- `WATCH` 不会直接下期货腿。
- 期权退出摩擦低时不生成 hedge intent。
- `HEDGE_READY` 必须同时满足风险、持续性、摩擦优势。

### Phase 5：小额实盘闸门

目标：

- 只在 Phase 1-4 全部空跑稳定后考虑。
- 先开期权开仓小额实盘，仍不开放自动 hedge。
- hedge 小额开关独立排期。

验收：

- 多时点回放通过。
- 真实空跑拒绝链一致。
- 账本重启恢复一致。
- maker-only 与价差守门有实盘证据。

## 10. 新增缺口分级判断

本节用于判断四个外部观点中，哪些必须进入近期主线，哪些允许作为后置缺口。判断标准不是“重要不重要”，而是：

```text
是否影响开闸前正确性？
是否影响当前方案的第一性边界？
是否影响执行链 polish 的必要闭环？
是否需要足够样本和实盘数据才能做得正确？
```

### 缺口 1：成本覆盖的全链端到端期望验证

观点：

```text
信号层单独前测，执行层单独计划，尚无 信号 -> spread -> Deribit 费率/价差/滑点 -> 净 P&L 的全链验证。
在该数字为正之前，不能谈长期盈利。
```

判定：**必须做，且是“实盘闸门前硬前提”。**

原因：

- 这是策略第一性里的收益真实性问题，不是体验问题。
- 信号层方向正确不等于执行结构扣完成本后为正。
- VRP 只回答“权利金是否足够补偿一部分波动风险”，也不能替代全链净 P&L。
- Deribit 期权费用、bid/ask、maker 挂单未成交、追价一步、保护腿成本、S:PM 释放，都会把纸面 edge 改写。

应该如何做：

```text
SignalReplayPackage
  -> ExecutionCandidateReplay
  -> VRP gate
  -> S:PM/cost/accounting simulator
  -> net_pnl_after_fee_spread_slippage
  -> attribution snapshot
```

最低验收：

- 至少输出 `gross_credit`、`fee`、`spread_cost`、`chase_cost`、`protection_cost`、`net_credit`、`expiry_or_exit_pnl`。
- 按信号强弱、方向、DTE、VRP gate、GGR regime 分桶。
- 不能只报胜率，必须报扣成本净期望、尾部亏损、回撤和样本数。
- 在该验证未完成前，文档只能说“执行链可控/可回放/可空跑”，不能说“长期盈利已成立”。

落地阶段：新增为 **Phase 2.5：全链成本回放**，位于 VRP 只读嵌入之后、小额实盘之前。

### 缺口 2：组合层 sizing 与区制条件化下注

观点：

```text
固定风险比例只是单笔口径。缺 Kelly/分数下注、并发持仓上限、跨仓 vega/gamma 预算、最大回撤熔断。
卖方必须在波动率扩张/趋势区制里缩量或离场，GGR 二元门不足以承担连续 sizing。
```

判定：**应该做，但分成两层：风险硬上限近期必须做，连续 sizing 可后置标定。**

必须近期做的部分：

- 并发持仓上限。
- 单标的最大 short gamma / short vega 上限。
- 账户级最大可用保证金占用。
- 日内/周内最大亏损熔断。
- 连续亏损或波动扩张状态下的最小缩量规则。
- `ALLOW_TRADING` 开闸前的 `PortfolioRiskBudgetPackage`。

允许后置的部分：

- Kelly 或分数 Kelly。
- 精细化 regime -> size 连续函数。
- 多策略之间的 vega/gamma 配额优化。
- 基于长期样本的动态下注曲线。

原因：

- 没有组合硬上限，哪怕单笔结构正确，也可能因并发和尾部共振失控。
- 但 Kelly/连续 sizing 在样本不足时容易伪精确；应先做保守预算，再用全链回放和实盘小额数据校准。

推荐第一版 sizing：

```text
base_size = fixed_small_size
if signal == weak: size *= 0.5
if VRP edge thin: size *= 0.5
if GGR adverse / trend regime / vol expansion: size = 0 or min(size, test_size)
if portfolio gamma/vega/margin budget exceeded: size = 0
if drawdown fuse active: size = 0
```

落地阶段：硬预算放入 **Phase 1/2 同步设计**；连续 sizing 放入 **Phase 5 后的 Calibration Phase**。

### 缺口 3：赢家管理 / 滚动策略

观点：

```text
现在只有尾部防守，没有何时止盈、何时滚动。
空 Gamma 策略不主动管理赢家，会把到期前 gamma 尾部白白敞着。
```

判定：**应该做，且“止盈/末日 Gamma 退出”应早于实盘放量；复杂滚动允许后置。**

必须近期做的部分：

- 赢家止盈：例如权利金已回吐到目标比例时退出。
- 末日 Gamma 风险退出：到期前若 gamma/触界概率升高，即使未亏损也要退出或不再持有。
- 时间止盈/止损：持仓到某个剩余 DTE 阈值必须复核。
- 保护腿复用前必须重新验证，不因已有库存而强行卖新 short。

允许后置的部分：

- 多轮滚动收益优化。
- 滚动到新 expiry 的自动候选搜索。
- 基于 VRP/term structure 的动态 roll schedule。
- 保护腿库存的跨周期最优摊销算法。

原因：

- “如何避免赢家变输家”是卖方策略的基本生存问题。
- 但“滚到哪里、怎么滚最优”需要更多成交和回放数据，不适合在第一版自动化。

推荐第一版管理状态：

```text
POSITION_MANAGE
  -> TAKE_PROFIT_READY
  -> TIME_EXIT_REVIEW
  -> GAMMA_DECAY_EXIT
  -> ROLL_CANDIDATE_DRYRUN
  -> EXIT_OR_REUSE
```

落地阶段：基础止盈/末日 Gamma 退出放入 **Phase 4：Manage/Hedge**；复杂滚动放入后置优化。

### 缺口 4：收益归因闭环

观点：

```text
没有收益来源归因，就分不清赚的是 theta、方向、波动率补偿、保护腿残值，还是运气。
冷启动若不带归因，等于盲跑。
```

判定：**必须做，但可以先做最小归因，不必一开始做完整研究平台。**

原因：

- 归因是校准信号、VRP、执行成本和管理策略的共同底座。
- 没有归因，全链回放即使净 P&L 为正，也不知道是哪个模块在付钱。
- VRP 的 edge 验证尤其依赖归因：要区分 theta、IV/RV、方向和运气。

第一版最小归因字段：

```json
{
  "theta_capture": 0,
  "directional_pnl": 0,
  "iv_rv_edge_proxy": 0,
  "fee_cost": 0,
  "spread_slippage_cost": 0,
  "protection_cost_or_recovery": 0,
  "margin_release_context": 0,
  "hedge_pnl": 0,
  "unexplained_residual": 0
}
```

必须近期做的部分：

- 每笔交易结束后写 `AttributionPackage`。
- 至少区分收入、费用、价差/滑点、保护腿成本/残值、方向 PnL、hedge PnL。
- 与 `ExecutionSessionPackage` 绑定。

允许后置的部分：

- 精细 Greeks 级别归因。
- 波动曲面分解。
- 多日滚动归因图表。
- 归因驱动的自动参数优化。

落地阶段：最小归因应与 **Phase 3：EntryRiskAnchor 与账本增强** 同步；高级归因后置。

### 四个缺口的最终优先级

| 缺口 | 判定 | 是否开闸前必须 | 允许后置部分 |
| --- | --- | --- | --- |
| 全链端到端扣成本期望 | 必须做 | 是 | 只有更复杂的多因子研究可后置 |
| 组合 sizing / 区制下注 | 部分必须 | 硬预算必须；连续函数可后置 | Kelly、动态 sizing、组合优化 |
| 赢家管理 / 滚动 | 部分必须 | 止盈和末日 Gamma 退出必须 | 复杂自动 roll |
| 收益归因闭环 | 必须做最小版 | 是 | 高级 Greeks/曲面归因 |

结论：

```text
不能后置到实盘之后的：
1. 全链扣成本净 P&L 回放
2. 组合硬风险预算
3. 基础止盈/末日 Gamma 退出
4. 最小收益归因

允许后置到数据积累后的：
1. Kelly / 连续 sizing 函数
2. 多仓组合优化
3. 自动滚动策略
4. 高级 Greeks 与波动曲面归因
```

对当前整合方案的影响：

- `ExecutionSession v0.5` 仍是正确主线。
- 但它不能只解决 PLAN/ORDER polish；必须在小额实盘前补上 `ReplayExpectationPackage`、`PortfolioRiskBudgetPackage`、`AttributionPackage` 和基础 winner management。
- 换句话说，polish 不是把按钮做顺，而是让“信号、成本、仓位、退出、归因”成为同一条执行会话里的可审计事实。

## 11. 关键不变量

1. 信号层不持钥、不下单、不选腿。
2. VRP 不进信号层，不降低方向门槛。
3. VRP 是硬门，不是软权重。
4. 执行层不重判方向。
5. 执行层每次执行前必须重取价、重验 VRP、重验 S:PM、重验账本。
6. 任何 approval 都有 TTL，过期不可执行。
7. 任何方案都用 `plan_hash` 防止重排误选。
8. 对冲只在持仓后运行，不参与入场。
9. 对冲实盘开关独立于期权开仓开关。
10. 所有模块通过版本化包互联，不互相读内部状态。

## 13. 本稿最终建议

当前不应追求“把所有模块合成一个程序”，而应追求“把所有模块合成一个通顺的执行会话”。

最小正确推进顺序：

1. 先定 `ExecutionSessionPackage` 与 `ApprovalIntentPackage`。
2. 再改执行层用户流程，消灭重启式 PLAN/ORDER 体验。
3. 再把 VRP 作为 PLAN 内只读硬门接入。
4. 再把对冲作为持仓管理段接入。
5. 最后才谈小额实盘开关。

这条路线不破坏第一性：信号仍是信号，VRP 仍是定价门，执行仍是执行，对冲仍是持仓后风险压缩。它也能解决 polish 问题：用户面对的是一个从信号到计划、授权、执行、记账、管理、对冲的连续 cockpit，而不是一组需要重启和手填编号的脚本。
