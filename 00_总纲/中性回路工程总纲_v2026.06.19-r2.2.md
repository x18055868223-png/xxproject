## 3. 模块收纳规则

00-04 的稳定惯例：

- 每个编号模块有 `因子文档/`，从 `00_...总览.md` 开始。
- 运行/设计资产放在 `交付物快照/` 或明确的部署/源码目录中。
- 文档开头先给当前口径与历史边界。
- 单因子文档使用“定位、实现、关键字段或阈值、整合路径、待办、边界”结构。

r2.2 对 05 和 `deploy/signal_audit` 的处理：

## 4. GEX Monitor API

入口：

```text
05_GEX监控API_数据增强接口/
05_GEX监控API_数据增强接口/因子文档/00_GEX监控API总览.md
```

当前职责：

- 提供 `/health`、`/v1/info`、`/v1/refresh`。
- 把 GEX Monitor 公开页面的 `gex_board`、`gamma_exposure`、`volatility`、`flow` 转为结构化 JSON。
- 在本地追加历史样本，计算 netGEX、DVOL、IV/RV、PCR、Call share、Flow P/C 等 rank。
- 输出 `sections`、`missing_fields`、`field_status`，让信号层和审计页能解释数据质量。

边界：

- rank 是当前样本窗口中的相对位置，不是胜率。
- `quality=warming_up` 表示样本不足或未满 30 天。
- GEX API 是只读上下文增强，不解锁交易门。

## 5. 审计静态页面与 materializer

入口：

```text
deploy/signal_audit/
deploy/signal_audit/因子文档/00_审计部署总览.md
deploy/signal_audit/docs/审计卡片语义.md
```

当前职责：

- `tools/materialize_signal_cards.py` 从 FMZ `signal_review.jsonl` 生成 `signal_cards/index.json`、单卡 JSON 和 `fallback.js`。
- `tools/gemini_signal_llm_review.py` 在策略外生成 LLM sidecar。
- `deploy/signal_audit/install_or_update.sh` 把静态页面和工具安装到策略服务器，并启用 materializer / LLM timer。
- `deploy/signal_audit/frontend/` 是提交到仓库的静态阅读器和样例资产。

边界：

- committed `signal_cards` 是 fixture，不是生产 JSONL 全量数据。
- `install_or_update.sh` 是主动审计服务更新，不是只读检查。
- LLM 复核不改变 `decision`、`reasoning`、`blocking`、`trade_allowed`。
- 真实 `GEMINI_API_KEY` 只放 `/etc/signal-audit/llm.env`。
